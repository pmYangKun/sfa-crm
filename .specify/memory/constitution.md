# SFA CRM 宪法

## 核心原则

### 一、Ontology 优先的数据模型
所有业务实体显式建模为**对象**、**关系**和**动作**，不采用关系表+外键的传统建模方式。每个对象有明确的生命周期和状态机。每次状态变更必须由显式动作（事件）触发，禁止直接修改字段。事件记录不可变，派生状态从事件历史计算得出。

*原因：客户归因、客保机制、商机阶段等业务规则复杂，平铺表结构会导致规则隐含在代码里，难以审计和维护。Ontology 让规则显式可查。*

### 二、API 优先，统一操作层
无论是人通过 GUI 发起的操作，还是 AI Agent 发起的操作，必须全部经过同一套 API 层。任何组件不得绕过 API 直接访问数据层。API 是所有操作权限和业务规则的唯一执行点。

*原因：Copilot/Agent 能力要求与人工操作执行同样的业务规则校验。如果存在"内部通道"，规则就会产生分歧。*

### 三、业务规则可配置，不硬编码
因区域、角色或策略不同而变化的业务规则（公共池抢占规则、客保期限、线索分配策略），必须存储为配置，不得嵌入代码。代码实现规则引擎，配置驱动规则内容。

*原因：客户现有5个大区规则各不相同，且规则会持续变化。硬编码意味着每次调整都要改代码。*

### 四、数据完整性不可妥协
客户唯一性必须在系统层面强制保证，不依赖 UI 校验。任何线索录入前必须完成去重检测。针对竞争性操作（公共池抢占）必须实施速率限制，防止自动化工具滥用。这是系统级保障，不是可选功能。

*原因：重复客户记录和外挂抢池是已记录的活跃问题，正在破坏数据质量。*

### 五、最小化销售录入负担
销售被要求手动录入的每一条数据，必须有明确的必要性依据。凡是可以推导或自动生成的内容（日报从拜访记录生成、商机状态从事件推导），必须自动化处理。系统应请销售确认，而不是重复录入。

*原因：销售对 CRM 的抵触与录入负担直接相关。低采纳率产生劣质数据，系统目标就会落空。*

### 六、显式优于隐式
每条业务规则、状态转换、权限检查和配置项，必须在 spec 中明确声明和描述。不存在"不言自明"的逻辑。隐式行为是潜在的 bug。

*原因：AI 辅助实现要求规格无歧义。模糊的 spec 会在不同会话中产生不一致的代码。*

## 技术约束

- **数据层**：基于 Ontology 建模；在编写任何 schema 之前，先完成对象和关系的显式定义
- **技术栈**：前端 Next.js / 后端 FastAPI（Python）/ 数据库 SQLite（演示用）/ 部署 Docker Compose + 腾讯云轻量服务器
- **API 层**：RESTful；GUI 和 Agent 共用同一套端点；在 API 层执行所有业务规则
- **AI Agent 层**：Vercel AI SDK 负责 LLM 交互；LLM Provider 存数据库配置，Admin 可切换（默认 Claude，支持 GPT/DeepSeek 等）；Ontology Actions 直接映射为 Tool Use 工具定义；Skill 系统以提示词文本存储于数据库，Admin 维护，LLM 通过工具调用检索
- **速率限制**：在 API 层实施，针对竞争性操作，客户端不可绕过
- **审计追踪**：所有动作（事件）仅追加，不对业务对象做物理删除
- **配置驱动**：大区规则、公共池策略、阈值参数、LLM Provider 配置存储在配置中，不写死在代码里
- **系统集成**：课时订单系统付款事件→CRM 商机状态更新（优先集成）；飞书通知（后续）；其余集成不在范围内

## 开发规范

- **先写 spec，再写代码**：每个功能在完成对应 spec 之前不得开始实现
- **唯一真相源**：业务逻辑只存在于 API 层；UI 和 Agent 是薄客户端
- **状态从事件推导**：派生状态（如是否在私有池、商机成熟度）始终计算得出，不作为可变字段存储
- **关键路径必须测试**：客户去重、池分配规则、速率限制、拆单识别是必测项
- **变更流程**：业务规则变更→先更新配置 spec→再更新 API→最后更新 UI/Agent

## 治理

本宪法凌驾于所有其他设计决策之上。当实现选择与原则冲突时，原则优先——实现必须调整。

**修订程序**：任何原则变更，必须先在宪法中记录讨论和理由，再反映到代码中。宪法版本号必须随之递增。

**版本策略**：
- MAJOR：原则被删除或根本性重定义
- MINOR：新增原则或约束
- PATCH：澄清说明、措辞修正
