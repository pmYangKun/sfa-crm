# SFA CRM — 功能规格（Specifications）

> 基于 `ontology-v0.2.md` 和 `business-context.md` 整理。
> 本文件描述系统需要实现的功能边界，不涉及技术实现方式。
> 每个 Feature 包含：用户故事、验收条件、边界情况。

---

## SPEC-001：线索录入与唯一性校验

### 用户故事

作为销售，我需要录入一条新线索，系统帮我判断这家企业是否已经存在，避免重复录入导致客保纠纷。

### 验收条件

1. 销售填写企业名称后，系统立即执行唯一性检测：
   - 若填写了组织机构代码：与库中所有线索/客户的代码做精确匹配；命中则阻断录入，提示"该企业已存在，当前归属：[销售姓名]"
   - 若未填写组织机构代码：用企业名称相似度算法做模糊检测；相似度超过阈值（可配置），生成疑似重复预警
2. 疑似重复预警推送给战队队长，队长决定是否合并
3. 队长确认为重复：新线索被驳回，提示"已有相同企业存在"
4. 队长确认为不同企业：新线索正常入库
5. 两个联系人的 wechat_id 或 phone 相同时，系统自动创建 ContactRelation 并通知队长
6. 线索录入后默认进入公共池（`pool = public`），需等待主管分配

### 边界情况

- 企业名称仅相差"有限公司"/"股份有限公司"等法律后缀时，视为相似，触发预警
- 同一销售在预警待处理期间，不得二次录入同名企业
- 组织机构代码填写后不可清空

## SPEC-002：线索分配（私有池管理）

### 用户故事

作为战队队长，我需要把公共池里的线索分配给我的销售，同时保证没有销售的私有池超过上限。

作为销售，我需要知道我的私有池还剩多少名额，避免在抢占时被系统拒绝。

### 验收条件

1. 主管（队长/大区总/admin）可将 `pool = public` 的线索分配给指定销售：
   - 前提：目标销售的私有池数量 < 上限（默认 100）
   - 执行后：`lead.owner = 目标销售`, `lead.pool = private`
2. 销售可在"我的私池"页看到当前数量和上限
3. 超出上限时，分配操作被阻断，提示"该销售私有池已满（当前 X / 上限 Y）"
4. 私有池上限为全局可配置项，仅 admin 可修改
5. 线索转移后，原销售**立即失去可见性**

### 边界情况

- 私有池数量只统计 `stage = active` 的线索，`converted` 和 `lost` 不占名额
- 分配操作必须有日志记录

## SPEC-003：公共池抢占

### 用户故事

作为销售，我需要从公共池里主动抢占线索，系统需要保证抢占公平，防止外挂。

### 验收条件

1. 销售可查看公共池线索列表（仅显示本大区的线索）
2. 销售点击"抢占"后，系统验证：
   - `lead.pool == public`
   - `sales.private_pool_count < pool_limit`
   - 当前大区抢占规则允许
   - 当前账号抢占频率未超限（每分钟操作次数 < N，N 可配置）
3. 验证通过：`lead.owner = 当前销售`, `lead.pool = private`
4. 超出频率限制：操作被拒绝，账号锁定，通知队长
5. 大区抢占规则由 admin 配置，支持三种模式：
   - 仅战队内可抢
   - 战队间可抢
   - 大区长助理手工分派（销售无法自行抢占）
6. 抢占操作有全量日志

### 边界情况

- 两个销售同时抢同一线索：先到先得，后者返回"该线索已被抢占"
- 账号锁定后由队长手动解锁，不自动恢复

## SPEC-004：线索自动释放

### 用户故事

作为系统，我需要自动把长期未跟进或未成单的线索释放回公共池，保证线索资源不被低效占用。

### 验收条件

1. 系统每日定时检测私有池线索（`stage = active`），满足以下任一条件自动释放：
   - `now - lead.last_followup_at > 10天`
   - `now - lead.created_at > 30天`
2. 释放执行：`lead.owner = null`, `lead.pool = public`
3. 释放时通知原销售："您名下的 [企业名] 因 [释放原因] 已自动释放回公共池"
4. 两个阈值（10天/30天）均为可配置项，仅 admin 可修改
5. 主管可手动触发释放，不受天数限制

### 边界情况

- `stage = converted` 或 `lost` 的线索不参与自动释放
- 释放当天若有跟进记录，跳过本次检测

## SPEC-005：线索转化为客户

### 用户故事

作为系统，当销售的线索购买了小课，我需要自动将其升级为客户，并完整保留历史跟进数据。

### 验收条件

1. 触发方式：
   - 优先：课时订单系统付款事件自动触发
   - 兜底：销售手动确认（需选择对应线索）
2. 执行：
   - `lead.stage = converted`, `lead.converted_at = now`
   - 创建 Customer 记录，`customer.lead_id = lead.id`，继承企业名称、组织机构代码、大区、归属销售、线索来源
   - 线索下的联系人、跟进记录、关键事件全部迁移到 Customer 下
3. 转化后线索从活跃列表消失，进入归档，仍可通过客户记录追溯
4. 转化完成后，销售收到通知："[企业名] 已转化为客户"

### 边界情况

- 同一线索不可重复转化
- 手动确认需要操作者为该线索的 owner 或主管

## SPEC-006：跟进记录

### 用户故事

作为销售，我需要记录每次与线索/客户的沟通，系统自动更新最近跟进时间，避免线索被释放。

### 验收条件

1. 销售可对自己名下的线索或客户新增跟进记录
2. 必填字段：跟进方式（电话/微信/拜访/其他）、跟进内容、实际发生时间
3. 可选关联具体联系人
4. 线索跟进提交后：`lead.last_followup_at` 更新为本次实际发生时间
5. 跟进记录不可删除，可追加补充说明
6. 主管可查看下属所有线索/客户的跟进记录

### 边界情况

- 实际发生时间可早于录入时间（允许补录）
- `followed_at` 不能晚于当前时间
- 线索/客户转移后，原销售历史跟进记录保留，但无法继续录入

## SPEC-007：关键事件记录

### 用户故事

作为销售，我需要把拜访决策人、送书、参加小课这类关键动作结构化记录，方便主管和系统准确判断进展。

### 验收条件

**拜访关键决策人（visited_kp）：**
1. 销售记录"拜访 KP"，关联具体联系人（需标记为决策人）
2. 创建 `visited_kp` KeyEvent

**送书（book_sent）：**
1. 仅限线索阶段可操作
2. 记录送出时间；销售后续可更新"是否回应/确认阅读"字段

**参加小课（attended_small_course）：**
1. 优先由课时订单系统自动触发；失败时销售手动确认
2. 触发 `convert_lead` Action（线索转化为客户）

**购买大课（purchased_big_course）：**
1. 优先由课时订单系统自动触发；销售可手动兜底
2. 仅限客户阶段可操作
3. 记录合同金额

### 边界情况

- 同一对象不可重复创建同类型 KeyEvent，但可更新已有事件字段
- 手动操作需记录操作人

## SPEC-008：大课转化窗口提醒

### 用户故事

作为战队队长，我需要系统在客户参加小课后自动倒计时提醒，避免错过 14 天转化窗口。

### 验收条件

1. 客户创建时，系统记录 `customer.created_at` 作为窗口起点
2. 窗口判断条件：`now - customer.created_at < 14天 AND 课时系统无大课订单`
3. 系统在第 7 天、第 12 天分别推送提醒给销售和队长
4. 第 14 天仍无大课订单：推送提醒"[企业名] 转化窗口已关闭"，不修改客户任何字段
5. 窗口关闭后，客户记录保持不变，销售仍可继续跟进

### 边界情况

- 14 天为固定值，当前版本不可配置
- 若第 14 天之后客户仍购买大课，销售可手动录入 `purchased_big_course` 事件，系统正常记录

## SPEC-009：联系人管理

### 用户故事

作为销售，我需要在线索/客户下维护多个联系人，标注决策人，并记录跨企业的人脉关系。

### 验收条件

1. 销售可在自己名下的线索/客户下新增联系人
2. 添加时系统检测 wechat_id/phone 是否与库中已有联系人重复：
   - 重复：自动创建 ContactRelation，通知涉及双方企业的队长
3. 销售可手动建立两个联系人之间的关系，类型：夫妻/亲属/合伙人/朋友
4. 建立关系时，若两家企业归属不同销售，触发归属冲突检测，通知相关主管

### 边界情况

- 线索转化为客户时，联系人随之迁移到客户下
- 线索/客户转移后，联系人随之转移，原销售不可编辑

---

## SPEC-010：日报自动生成与提交

### 用户故事

作为销售，跟进记录已经写了，日报应该自动生成，我只需要确认提交。

### 验收条件

1. 每日 18:00（可配置）汇总销售当天所有 FollowUp，生成日报草稿
2. 销售收到推送，可编辑草稿后提交
3. 提交时选择对象：战队队长（必选）+ 大区总（可选）
4. 当天无 FollowUp 记录时不生成草稿，20:00 推送提醒"今日尚未提交日报"
5. 主管可查看下属日报提交情况

### 边界情况

- 草稿生成后补录跟进记录，草稿自动更新
- 已提交日报不可修改，只可追加补充说明

---

## SPEC-011：数据可见性与权限

### 用户故事

作为各层级人员，我只能看到我负责范围内的线索和客户。

### 验收条件

1. 权限规则基于 OrgNode 树：
   - `sales`：仅可查看自己名下的线索和客户
   - `manager`：可查看自己所在 OrgNode 及所有子节点下的全部线索和客户
   - `admin`：查看全部，不受 OrgNode 限制
2. 上述规则适用于所有数据视图：列表、跟进记录、关键事件、日报
3. 线索/客户转移后，原销售**立即失去所有可见性**
4. API 层同样做权限校验，不可通过 URL 越权访问

### 边界情况

- 主管权限范围随 OrgNode 挂载位置变化
- admin 可查看全量操作日志

---

## SPEC-012：组织架构管理

### 用户故事

作为 admin，我需要维护公司组织架构，包括新增/调整大区、战队，以及管理用户挂载。

### 验收条件

1. admin 可新增/编辑/停用 OrgNode 节点（名称、类型、父节点）
2. admin 可将 User 挂载到指定 OrgNode，或调整挂载位置
3. OrgNode 停用时，检查该节点下是否有活跃线索/客户或用户，若有则阻断并提示
4. 用户角色由 admin 设置，不可自行修改
5. 层级无上限

### 边界情况

- 根节点只有一个，不可删除
- OrgNode 调整后，挂在该节点下的所有数据可见性随之变更

## 菜单结构

### 销售
```
├── 我的私池        ← 自己名下的活跃线索
├── 公共池          ← 可抢占的线索（仅本大区）
├── 我的客户        ← 自己名下的客户
└── 我的日报        ← 草稿 / 提交 / 历史
```

### 主管（队长 / 大区总 / VP，数据范围按 OrgNode 自动变化）
```
├── 数据概览        ← 各销售业绩、私有池使用率、转化漏斗
├── 团队线索        ← 本节点及子节点下所有线索
├── 公共池          ← 待分配线索，可手动分配给销售
├── 团队客户        ← 本节点及子节点下所有客户
└── 团队日报        ← 下属日报提交情况
```

### Admin
```
├── 全部线索
├── 全部客户
├── 组织管理        ← OrgNode 树 + 用户挂载
├── 系统配置        ← 私有池上限 / 释放阈值 / 抢占规则 / 日报推送时间
└── 操作日志
```

### 线索详情页
```
线索详情
├── 基本信息        ← 企业名 / 大区 / 来源 / 归属销售 / 当前状态
├── 联系人
├── 跟进记录
└── 关键事件        ← 送书 / 拜访KP 等
```

### 客户详情页
```
客户详情
├── 基本信息        ← 企业名 / 大区 / 来源 / 归属销售 / 转化时间
├── 联系人
├── 跟进记录
├── 关键事件        ← 购买大课等
├── 转化窗口        ← 仅窗口期内显示，倒计时 + 课时购买情况
└── 来源线索        ← 链接到归档的原始线索记录
```
