# SFA CRM — 功能规格（Specifications）

> 基于 `ontology-v0.1.md` 和 `business-context.md` 整理。
> 本文件描述系统需要实现的功能边界，不涉及技术实现方式。
> 每个 Feature 包含：用户故事、验收条件、边界情况。

## SPEC-001：企业录入与唯一性校验

### 用户故事

作为销售，我需要录入一家新企业，系统帮我判断这家企业是否已经存在，避免重复录入导致客保纠纷。

### 验收条件

1. 销售填写企业名称后，系统立即执行唯一性检测：
   - 若填写了组织机构代码：与库中所有企业的代码做精确匹配；命中则阻断录入，提示"该企业已存在，当前归属：[销售姓名]"
   - 若未填写组织机构代码：用企业名称相似度算法做模糊检测；相似度超过阈值（可配置），生成疑似重复预警
2. 疑似重复预警推送给企业所在大区的战队队长，队长决定是否合并
3. 队长确认为重复时，新录入企业被驳回，提示销售"已有相同企业存在"
4. 队长确认为不同企业时，新企业正常入库
5. 两个联系人的 wechat_id 或 phone 相同时，系统自动创建 ContactRelation 并通知战队队长
6. 企业录入后默认进入公共池（`pool = public`），需等待主管分配

### 边界情况

- 企业名称仅相差"有限公司"/"股份有限公司"等法律后缀时，视为相似，触发预警
- 同一销售在预警待处理期间，不得二次录入同名企业
- 组织机构代码字段不强制（无法获取时可跳过），但填写后不可清空

## SPEC-002：企业分配（私有池管理）

### 用户故事

作为战队队长，我需要把公共池里的企业分配给我的销售，同时保证没有销售的私有池超过上限。

作为销售，我需要知道我的私有池还剩多少名额，避免在抢占时被系统拒绝。

### 验收条件

1. 主管（队长/大区总/admin）可将 `pool = public` 的企业分配给指定销售：
   - 前提：目标销售的私有池数量 < 上限（默认 100，不含 `converted` 状态企业）
   - 执行后：`company.owner = 目标销售`, `company.pool = private`
2. 销售可在"我的企业"页看到私有池当前数量和上限
3. 超出上限时，分配操作被阻断，提示主管"该销售私有池已满（当前 X / 上限 Y）"
4. 私有池上限为全局可配置项，仅 admin 可修改
5. 企业一旦进入私有池，原销售可见，转移后原销售**立即失去可见性**

### 边界情况

- `converted` 状态的企业不占私有池名额（已成单，不再参与抢占竞争）
- 分配操作必须有日志记录（谁在什么时间把哪家企业分给了谁）

## SPEC-003：公共池抢占

### 用户故事

作为销售，我需要从公共池里主动抢占有意向的企业放入我的私有池，系统需要保证抢占公平，防止外挂。

### 验收条件

1. 销售可查看公共池企业列表（仅显示本大区的企业）
2. 销售点击"抢占"后，系统验证：
   - `company.pool == public`
   - `sales.private_pool_count < pool_limit`
   - 当前大区抢占规则允许（见规则配置）
   - 当前账号抢占频率未超限（每分钟操作次数 < N，N 可配置）
3. 验证通过：`company.owner = 当前销售`, `company.pool = private`
4. 超出频率限制：操作被拒绝，账号锁定，通知战队队长
5. 大区抢占规则由 admin 配置，支持三种模式：
   - 仅战队内可抢（默认）
   - 战队间可抢
   - 大区长助理手工分派（销售无法自行抢占）
6. 抢占操作有全量日志，包含时间戳和操作账号

### 边界情况

- 两个销售同时抢同一企业：先到先得，后者返回"该企业已被抢占"
- 账号锁定后，由队长手动解锁，不自动恢复

## SPEC-004：企业自动释放

### 用户故事

作为系统，我需要自动把长期未跟进或未成单的企业释放回公共池，保证线索资源不被低效销售占用。

### 验收条件

1. 系统每日定时检测私有池企业，满足以下任一条件自动触发释放：
   - `now - company.last_followup_at > 10天`（未跟进）
   - `now - company.created_at > 30天 AND company.stage == lead`（未成单）
2. 释放执行：`company.owner = null`, `company.pool = public`
3. 释放时通知原销售："您名下的 [企业名] 因 [释放原因] 已自动释放回公共池"
4. 两个释放阈值（10天/30天）均为可配置项，仅 admin 可修改
5. 主管也可手动触发释放，不受天数限制

### 边界情况

- `stage == customer` 或 `converted` 的企业**不参与自动释放**（已购课客户不因未跟进被释放）
- 释放当天若有跟进记录，跳过本次检测（`last_followup_at` 已更新）

## SPEC-005：跟进记录

### 用户故事

作为销售，我需要记录每次与客户的沟通，系统自动更新最近跟进时间，避免我的企业被系统释放。

### 验收条件

1. 销售可对自己名下的企业（`company.owner == current_user`）新增跟进记录
2. 跟进记录必填字段：跟进方式（电话/微信/拜访/其他）、跟进内容、实际发生时间
3. 可选关联具体联系人（Contact）
4. 提交后：`company.last_followup_at` 更新为本次实际发生时间
5. 跟进记录不可删除，可追加补充说明
6. 主管可查看下属所有企业的跟进记录

### 边界情况

- 实际发生时间可以早于录入时间（允许补录）
- 未来预约时间不允许（`followed_at` 不能晚于当前时间）
- 企业转移后，原销售的历史跟进记录仍保留，但原销售无法继续录入

## SPEC-006：关键事件记录

### 用户故事

作为销售，我需要把拜访决策人、送书、参加小课这类关键动作结构化记录，而不是混在普通跟进备注里，方便主管和系统准确判断商机阶段。

### 验收条件

**拜访关键决策人（visited_kp）：**
1. 销售记录本次跟进为"拜访 KP"，关联具体联系人（需标记为 `is_key_decision_maker = true`）
2. 创建 `visited_kp` KeyEvent，`payload.contact_id = 被拜访联系人`

**送书（book_sent）：**
1. 销售触发送书动作，前提：`company.stage == lead`
2. 记录送出时间 `payload.sent_at`
3. 销售后续可更新"客户是否回应/确认阅读"字段（`payload.responded_at`, `payload.confirmed_reading`）

**参加小课（attended_small_course）：**
1. 优先由课时订单系统付款事件自动触发；若自动触发失败，销售可手动确认
2. 前提：`company.stage == lead`
3. 执行后：`company.stage = customer`，创建 `attended_small_course` KeyEvent，**开启14天大课转化窗口**

**购买大课（purchased_big_course）：**
1. 优先由课时订单系统自动触发；销售可手动兜底
2. 前提：`company.stage == customer`
3. 执行后：`company.stage = converted`，创建 `purchased_big_course` KeyEvent，关闭转化窗口
4. 记录合同金额 `payload.contract_amount`

### 边界情况

- 同一企业不可重复创建同类型 KeyEvent（如不能送两次书），但可更新已有事件的字段
- 小课和大课的手动确认需要主管在场或有单据佐证（业务层约束，系统记录操作人）

## SPEC-007：大课转化窗口

### 用户故事

作为战队队长，我需要系统在客户参加小课后自动倒计时，并在窗口快关闭时提醒我和销售，避免错过转化机会。

### 验收条件

1. 企业进入 `customer` 阶段时，系统自动记录窗口开启时间（来自 `attended_small_course.occurred_at`）
2. 窗口期内（14天）：系统在第7天、第12天分别推送提醒给销售和战队队长
3. 第14天未出现 `purchased_big_course` 事件：`company.stage = lost`，通知销售和队长"转化窗口已关闭"
4. 14天为固定值，不可配置（当前版本）
5. 窗口关闭后企业进入 `lost` 状态，但所有历史记录保留，销售仍可查看

### 边界情况

- 转化窗口关闭后，若客户后续仍购大课，销售可手动触发 `confirm_big_course`，状态从 `lost` 恢复为 `converted`，并记录特殊说明

## SPEC-008：联系人管理

### 用户故事

作为销售，我需要在企业下维护多个联系人，标注决策人，并记录跨企业的人脉关系，避免因信息不透明导致销售策略冲突。

### 验收条件

1. 销售可在自己名下企业下新增联系人，字段：姓名、职位、是否决策人、微信号、手机号
2. 添加联系人时，系统检测 wechat_id/phone 是否与库中已有联系人重复：
   - 重复：自动创建 ContactRelation，推送通知给涉及双方企业的战队队长
3. 销售可手动建立两个联系人之间的关系（`link_contacts`），类型：夫妻/亲属/合伙人/朋友
4. 建立关系时，系统检测两家企业是否归属不同销售，若是则触发归属冲突检测，通知相关主管

### 边界情况

- 联系人归属企业转移后，该联系人随企业一起转移，原销售不可编辑
- 同一企业下同名联系人允许存在（同名不同人），以 wechat_id/phone 区分

## SPEC-009：日报自动生成与提交

### 用户故事

作为销售，我不想重复填报——跟进记录已经写了，日报应该自动生成，我只需要确认一下就提交。

### 验收条件

1. 每日定时任务（晚上18:00，可配置）汇总销售当天的所有 FollowUp 记录，生成日报草稿
2. 销售收到推送："您的日报草稿已生成，请确认后提交"
3. 销售可编辑草稿内容（在系统生成基础上补充说明），然后提交
4. 提交时，销售选择提交对象：战队队长（必选，不可取消）+ 大区总（可选，勾选后提交）
5. 若当天无 FollowUp 记录，不生成草稿，但系统在20:00推送提醒"今日尚未提交日报"
6. 主管可查看下属的日报提交情况（是否提交、提交时间）

### 边界情况

- 日报草稿生成后，若销售在提交前又补录了跟进记录，草稿自动更新（或提示销售刷新）
- 已提交的日报不可修改，只可追加补充说明

## SPEC-010：数据可见性与权限

### 用户故事

作为各层级管理人员，我只能看到我负责范围内的企业和数据，不该看的不显示。

### 验收条件

1. 权限规则基于 OrgNode 树：
   - `sales`：仅可查看 `company.owner == self` 的企业
   - `manager`（含队长/大区总/VP）：可查看自己所在 OrgNode 及所有子节点下的全部企业
   - `admin`：查看全部企业，不受 OrgNode 限制
2. 上述规则适用于所有数据视图：企业列表、跟进记录、关键事件、日报
3. 企业转移后，原销售**立即失去所有可见性**（列表消失，直接链接返回无权限）
4. admin 可查看全量操作日志

### 边界情况

- 主管查看范围随 OrgNode 挂载位置决定：若大区总被调到其他大区，权限随之变更
- 系统只做显示层过滤，API 层同样需要做权限校验（不能通过 URL 直接访问越权数据）

## SPEC-011：组织架构管理

### 用户故事

作为 admin，我需要维护公司的组织架构，包括新增/调整大区、战队，以及把销售挂到正确的节点下。

### 验收条件

1. admin 可新增/编辑/停用 OrgNode 节点（名称、类型、父节点）
2. admin 可将 User 挂载到指定 OrgNode，或调整挂载位置
3. OrgNode 停用时，系统检查该节点下是否有活跃企业或用户，若有则阻断并提示
4. 用户角色（`sales` / `manager` / `admin`）由 admin 设置，不可自行修改
5. 层级无上限，系统不对层级数量做约束

### 边界情况

- 根节点（root）只有一个，不可删除
- OrgNode 调整（如战队从大区A移到大区B）后，挂在该节点下的所有企业可见性随之变更
