# Feature Specification: 登录页改造 + 移动端 + Onboarding（UX 版）

**Feature Branch**: `001-login-mobile-onboarding`
**Created**: 2026-05-03
**Status**: Draft
**Input**: 完整业务决策来源 — `specs/001-login-mobile-onboarding/inputs/alignment.md`
**Scope Note**: 本 feature 是**纯 UX 增量**。公网部署、数据隔离、API 限流、Prompt Injection 防护等均**显式排除**，留待下一发版。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - PC 端访客试玩闭环（Priority: P1）

公众号读者点击 PC 端链接 → 看到 Hero 标题 + 双角色卡片 + 项目亮点三段（VibeCoding × Spec Coding / Palantir Ontology / Copilot 人机协同）→ 选一个角色卡片**一键登录**（无需输入账号密码）→ 进入 Dashboard，顶部出现"AI 引导区"展示推荐问题卡片 → 点击任一卡片，问题自动发送到 Chat 并执行 → 看到 AI 回答。

**Why this priority**: 这是项目"奠基阶段"对外展示的核心入口，决定首批读者对"Native AI CRM"的第一印象。如果 PC 闭环跑不通，整个增量失去意义。

**Independent Test**: 在 PC 浏览器打开站点 → 不输入任何账号密码 → 点 sales01 卡片 → 确认进入 Dashboard 看到引导区 → 点"自然语言查线索"卡片 → 确认问题发到 chat 并收到 AI 回答。

**Acceptance Scenarios**:

1. **Given** 访客首次访问 PC 登录页，**When** 页面加载完成，**Then** 屏幕呈现 Hero 主标题、两张角色卡片（销售·王小明 / 销售经理·陈队长）和三段项目亮点（含 emoji 图标）。页面**不出现**手动账号密码输入框。
2. **Given** 访客在 PC 登录页，**When** 点击"销售·王小明"卡片，**Then** 系统自动以 sales01 身份登录，跳转到 Dashboard，Dashboard 顶部展示"AI 引导区"含 4 张推荐卡片。
3. **Given** 已登录的 sales01 在 Dashboard，**When** 点击"自然语言查线索"引导卡片，**Then** 卡片对应的完整问题被自动发送到 Chat 并触发 AI 回答，引导区不消失。
4. **Given** 访客在 PC 登录页，**When** 点击"销售经理·陈队长"卡片，**Then** 以 manager01 身份登录并进入对应引导。
5. **Given** PC Dashboard 引导区，**When** 访客点击"切换销售/管理视角"卡片，**Then** 弹确认框 → 确认后自动 logout 当前账号 → 自动以目标角色登录 → 进入目标角色引导。

### User Story 2 - 移动端访客试玩闭环（Priority: P1）

公众号读者用手机点击链接 → 看到移动端登录页（Hero + 垂直堆叠的两张角色卡片 + 三段精简亮点）→ 点角色卡片一键登录 → 默认进入**全屏 Chat 模式**，Chat 顶部展示推荐问题卡片 → 点卡片体验 demo。底部金刚区可切换"线索 / 客户 / 💬 / 跟进 / 我的"。

**Why this priority**: 公众号读者最大入口是手机。移动端如果不做或做得差，公开发布的传播效果直接归零。

**Independent Test**: 用手机浏览器（或 PC 浏览器开发者工具切到 375px 视口）打开站点 → 点角色卡片 → 确认默认进入全屏 Chat → 确认看到引导卡片 → 点击卡片确认问题发到 chat 并收到 AI 回答 → 点底部金刚区"线索"切换到 Lead 列表。

**Acceptance Scenarios**:

1. **Given** 访客在移动端打开登录页，**When** 页面加载完成，**Then** 呈现 Hero 标题、垂直堆叠的两张角色卡片、三段一行核心句的项目亮点（无 GitHub 链接）。
2. **Given** 移动端访客，**When** 点击角色卡片，**Then** 自动登录并默认进入**全屏 Chat 模式**（金刚区中间💬按钮处于激活态）。
3. **Given** 移动端 Chat 全屏首屏（无消息），**When** 屏幕呈现，**Then** 顶部显示该角色对应的所有推荐卡片，输入框在底部不被遮挡。
4. **Given** 移动端 Chat 已有对话历史，**When** 用户发新消息，**Then** 引导卡片折叠隐藏（不再占用顶部空间）。
5. **Given** 移动端任意页面，**When** 点击金刚区任一 tab（线索 / 客户 / 跟进 / 我的），**Then** 切换到对应模块；点击中间💬，回到全屏 Chat。
6. **Given** 移动端用户在"我的" tab，**When** 点击"切换角色"，**Then** 弹确认 → 自动 logout + 以目标角色 login → 进入目标角色的引导（无需重新输密码）。

### User Story 3 - 移动端 Copilot 写类操作（Chat 内嵌卡片）（Priority: P1）

移动端用户在 Chat 输入"我今天拜访了深圳前海微链的赵鹏，聊了合作方案，对方很感兴趣。帮我把客户、联系人、跟进都录进去。" → AI 不直接写库，而是在 chat 流里**渲染 3 张待确认表单卡片**（客户 / 联系人 / 跟进，每张显示对象类型 + 关键字段预览）→ 用户**点任一卡片** → 屏幕底部弹起**表单抽屉** → 用户检查 / 修改字段 → 点"确认提交" → 抽屉关闭，对应卡片状态变为"✅ 已创建"并显示对象 ID。

**Why this priority**: 这是"Native AI CRM"的核心证据。PC 端 Copilot 是"AI 引导你填表"，移动端必须找到等价的、对触屏友好的范式，否则移动端只是 PC 的退化版。这条路径走通，才能在传播时说"AI 真的在替你做事"。

**Independent Test**: 在移动端 Chat 输入完整的多对象创建请求 → 确认 chat 流里出现 3 张待确认卡片（不是直接成功消息）→ 点其中一张 → 确认底部弹出表单抽屉显示预填字段 → 改一个字段 → 点确认 → 确认卡片状态变为已创建并显示 ID。

**Acceptance Scenarios**:

1. **Given** 移动端 Chat，**When** 用户输入触发多对象创建的请求，**Then** AI 回复中**不直接写库**，而在消息流中渲染对应数量的"待确认"表单卡片，每张卡片显示对象类型徽章 + 关键字段摘要。
2. **Given** chat 内嵌"待确认"卡片，**When** 用户点击卡片，**Then** 屏幕底部弹起表单抽屉，抽屉内容为该对象的完整可编辑表单且字段已被 AI 预填。
3. **Given** 表单抽屉打开中，**When** 用户修改任一字段并点"确认提交"，**Then** 抽屉收起，对应卡片状态变为"✅ 已创建"并展示创建后的对象 ID/链接。
4. **Given** 表单抽屉打开中，**When** 用户向下滑动 / 点关闭，**Then** 抽屉收起，对应卡片回到"待确认"态（不丢失 AI 预填值）。
5. **Given** 多张待确认卡片同屏纵向堆叠，**When** 用户分别处理（确认 / 编辑 / 暂留），**Then** 每张卡片状态独立互不影响。
6. **Given** 用户已确认部分卡片后继续在 chat 追问"再加一条备注"，**When** AI 响应，**Then** AI 可在 chat 流后续追加新的待确认卡片，不影响已确认卡片的状态。

### User Story 4 - PC 端 Copilot 兼容性保留（Priority: P2）

现有 PC Copilot 的"chat 右侧 + 表单左侧"双栏交互在本次改造后**继续可用**（即用户在 PC 端的 Chat 中说"录入跟进"，PC 端表现仍然是导航到左侧表单页并预填，不应被移动端的 chat 内嵌卡片范式取代）。

**Why this priority**: 移动端写类工具改造（"返回待确认对象而非直接写库"）会触及现有 PC Copilot 的 tool 调用层。如果架构抽象不当，可能把 PC 既有路径搞坏。这条故事确保 PC 兼容性不被破坏。

**Independent Test**: 在 PC 浏览器以 sales01 登录 → 在 chat 输入"我今天拜访了赵鹏…帮我把客户、联系人、跟进都录进去" → 确认 PC 端表现是**导航到左侧客户创建表单并预填**（而非 chat 内嵌卡片）。

**Acceptance Scenarios**:

1. **Given** PC 端 Copilot，**When** 用户输入触发写类操作的请求，**Then** PC 端表现保持**导航 + 表单预填**模式，不展示移动端的 chat 内嵌卡片。
2. **Given** PC 表单预填后，**When** 用户在表单页提交，**Then** 行为与本次改造前一致。

### Edge Cases

- **登录态丢失**：访客 cookie 过期或被清后回到任一内页 → 应自动重定向到登录页（PC / 移动各自的登录页），不能白屏。
- **角色切换失败**：一键切换过程中网络抖动导致 logout 成功但 login 失败 → 应回退到登录页并显示友好提示，不能卡在中间态。
- **AI 预填字段缺失或不完整**：AI 解析用户自然语言时无法提取所有必填字段 → 待确认卡片仍要显示，缺失字段在抽屉中以红色标记并允许用户补充。
- **移动端横屏**：移动端用户横屏时金刚区不应错位 / 不应遮挡内容；本次最低保证竖屏体验，横屏不做特殊优化但不可崩溃。
- **极小屏（< 360px）**：iPhone SE 等老机型仍要可用，垂直堆叠卡片不溢出。
- **多对象提交时部分失败**：用户依次确认 3 张卡片，第 2 张提交失败（如服务端校验拒绝）→ 第 1 张已创建结果保留，第 2 张卡片显示错误提示，第 3 张状态不受影响。
- **引导卡片在 chat 已有消息时再次出现**：用户清空对话或新开会话回到无消息态 → 引导卡片应再次出现（demo 性质，无"老用户"概念）。
- **直接深链进入内页**：访客通过分享链接直接进入某条 lead 详情页 → 未登录则重定向登录；已登录但角色不匹配 DataScope → 显示"无权访问"友好页。

## Requirements *(mandatory)*

### Functional Requirements

**FR-1 系列：PC 登录页**

- **FR-001**：PC 登录页 MUST 仅提供"角色卡片一键登录"入口，**不包含**手动账号密码输入框。
- **FR-002**：PC 登录页 MUST 至少展示两个角色卡片：销售·王小明（对应账号 sales01）和销售经理·陈队长（对应账号 manager01）。
- **FR-003**：PC 登录页 MUST 包含 Hero 标题 "这是一个用 AI 从零写到上线的 Native AI CRM"。
- **FR-004**：PC 登录页 MUST 包含项目亮点区，含三段方法论（VibeCoding × Spec Coding / Palantir Ontology / Copilot 人机协同），每段含图标 + 标题 + 1-3 行说明。
- **FR-005**：PC 登录页 MUST 在底部展示项目数据（Phase 数、commit 数）+ GitHub 源码地址 + 公众号文字引导（**不放二维码**）。
- **FR-006**：点击任一角色卡片 MUST 在 5 秒内完成自动登录并跳转到对应主页。

**FR-2 系列：移动端登录页**

- **FR-007**：移动端登录页 MUST 复用 PC 登录页的角色卡片入口（同样仅卡片登录），但布局为**垂直堆叠**。
- **FR-008**：移动端登录页 MUST 提供项目亮点的精简版（三段每段 1 行核心句 + emoji 图标），**不包含 GitHub 链接**。
- **FR-009**：移动端登录页 MUST 在 320px - 480px 视口内完整显示无横向滚动。

**FR-3 系列：移动端 Layout**

- **FR-010**：移动端 MUST 提供独立的 Mobile Layout（含底部金刚区导航 + 全屏内容区），与 PC Desktop Layout（侧边栏 + 右侧 chat）**互不污染**。
- **FR-011**：金刚区 MUST 含 5 个 tab：线索 / 客户 / 💬 AI 助手 / 跟进 / 我的；中间💬 tab 为视觉凸起按钮。
- **FR-012**：登录后 MUST 默认进入 Chat 全屏（💬 tab 激活态）。
- **FR-013**："我的" tab MUST 提供：账号信息展示 + **一键切换角色**（sales01 ↔ manager01）+ 退出登录。
- **FR-014**：一键切换角色 MUST 无需用户重新输入账号密码，全流程在 3 秒内完成。

**FR-4 系列：首次登录引导**

- **FR-015**：PC 端 MUST 在 Dashboard 顶部展示"AI 引导区"，含欢迎语 + 推荐卡片网格 + 切换角色提示。引导区下方保留正常 Dashboard 数据。
- **FR-016**：移动端 MUST 在 Chat 全屏的无消息态展示推荐卡片；有消息后引导卡片折叠隐藏。
- **FR-017**：引导卡片清单（每张含短标题 emoji + 完整问题预览 + "试试看 →"按钮）：
  - sales01：① 🔍 自然语言查线索 / ② ⚡ 一句话搞定多个动作 / ③ 🧠 让 AI 给我跟进策略 / ④ 💡 切换 manager01
  - manager01：① 🚨 谁在偷懒？AI 一眼看穿 / ② 🔬 评估具体线索跟进情况 / ③ 💡 切换 sales01
- **FR-018**：引导卡片 PC 端和移动端**全部展示**（移动端裁剪策略推后到上线后看数据决定）。
- **FR-019**：引导卡片点击 MUST 自动把"完整问题预览"作为消息发送到 Chat 并触发 AI 执行，复用现有 sessionStorage 模式。
- **FR-020**："切换角色"卡片点击 MUST 弹确认对话框；确认后自动 logout + login 目标角色 + 进入目标引导。
- **FR-021**：引导卡片 MUST **永远显示**（无"已读"持久化），demo 性质要求每次进入 chat 无消息态都能看到。

**FR-5 系列：移动端 Copilot 写类操作（Chat 内嵌卡片）**

- **FR-022**：移动端 Chat 中触发写类操作时，AI MUST 不直接写入数据库，而是返回结构化"待确认对象"列表。
- **FR-023**：每个"待确认对象" MUST 在 chat 流中渲染为一张表单预览卡片，含：对象类型徽章、关键字段预览（≥ 2 个）、状态徽章（待确认 / 编辑中 / 已创建 / 失败）。
- **FR-024**：点击预览卡片 MUST 从屏幕底部弹起表单抽屉，抽屉内容为该对象的完整可编辑表单（复用现有 PC 表单字段组件），AI 预填的字段已填好。
- **FR-025**：抽屉内"确认提交" MUST 调用对应的真实写类接口完成持久化；成功后抽屉收起、卡片状态切到 ✅ 已创建并显示对象 ID/链接。
- **FR-026**：抽屉关闭（向下滑 / 点 X）MUST 不丢失 AI 预填值，卡片回到"待确认"态。
- **FR-027**：多张待确认卡片状态 MUST 相互独立，部分失败不影响其他。
- **FR-028**：PC 端 Copilot 行为 MUST 保持现状（chat 中提到写类时仍走"导航到左侧表单页 + 预填"），不被本次移动端范式改造干扰。

**FR-6 系列：跨端共享**

- **FR-029**：PC 与移动端 MUST 共用同一份业务页面代码（Lead 列表 / Customer 列表 / 跟进流等），仅 Layout 和登录页是两套。
- **FR-030**：PC 与移动端 MUST 共用同一份 demo 案例数据 / system prompt / 工具调用层（不为移动端 fork）。

### Key Entities

- **Demo 账号 (Demo Account)**：固定的演示账号集合（sales01 王小明 / manager01 陈队长 / 其他既有账号）。每个含账号名、显示名、对应 DataScope、角色描述（用于登录页卡片副本）、对应引导卡片清单。
- **角色卡片 (Role Card)**：登录页入口元素。属性：对应账号、显示名、角色一句话描述（如"一线销售视角：用 Chat 跟进客户、查询线索、自动预填表单"）。PC / 移动两端共用数据，布局不同。
- **引导卡片 (Onboarding Card)**：登录后呈现的推荐问题卡片。属性：所属角色、平台支持（PC / Mobile / Both）、短标题（含 emoji）、完整问题预览（即点击后发到 chat 的真实 prompt）、对应 demo case 编号、卡片类型（demo 问题 / 角色切换）。
- **Chat 表单卡片 (Chat Form Card)**：仅移动端使用。AI 写类操作的 chat 内嵌待确认对象。属性：对象类型（客户 / 联系人 / 跟进 / 任务等）、AI 预填字段集合、当前状态（pending / editing / submitted / failed）、提交后的对象 ID（成功后才有）、错误信息（失败后才有）。
- **金刚区 Tab (KingKong Tab)**：仅移动端使用。属性：图标、标题、对应路由、是否凸起（中间💬唯一为 true）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

**SC-001**：访客从 PC 登录页加载完成到点击角色卡片登录后看到 Dashboard 引导区，在 5 秒内完成（含网络往返、不含 AI 首字 token）。

**SC-002**：移动端从登录页加载完成到点击角色卡片登录后看到 Chat 全屏 + 引导卡片，在 5 秒内完成。

**SC-003**：访客在移动端 320px - 480px 视口下访问任一页面（登录 / Chat / Lead 列表 / Customer 列表 / 跟进 / 我的），**无横向滚动条**。

**SC-004**：访客在不读任何说明的情况下，从打开链接到看到 AI 第一条有意义的回答，全流程可在 60 秒内完成（PC 与移动两端各自独立达标）。

**SC-005**：移动端 Chat 内嵌表单卡片范式可成功完成多对象批量创建——访客发出多对象请求后，能在 chat 流中看到对应数量的预览卡片，且至少能完整提交其中一张并看到"已创建"状态。

**SC-006**：一键切换角色全流程（点切换按钮 → 确认 → logout → login → 进入新角色引导）≤ 3 秒。

**SC-007**：PC 端 Copilot 现有行为（chat 右侧 + 表单左侧）在改造后不发生回归——既有 PC 写类用例 100% 仍按"导航 + 预填"模式工作。

**SC-008**：项目亮点区文案可在不查文档的前提下让一位 B 端 PM 读者在 30 秒内说出本项目"和普通 AI 项目的区别在于"什么。

## Assumptions

- **既有数据**：sales01 / manager01 等 demo 账号已在数据库 seed 中存在，可直接用于登录；密码为既有 demo 默认值（前端硬编码用于一键登录）。
- **既有功能**：PC 端现有 chat-sidebar 的 sessionStorage 导航预填机制已稳定可用，移动端引导卡片可直接复用同一模式发送问题到 chat。
- **既有工具层**：现有 AI 写类工具（创建客户 / 联系人 / 跟进等）的入参 schema 已稳定，可在其上加一层"返回待确认对象"的封装而不重写底层逻辑。
- **响应式策略**：业务页面（Lead 列表、Customer 列表、跟进流等）通过 Tailwind 响应式类即可在手机视口下基本可用；如个别业务页面体验差，本次只兜底"可读不崩溃"，不做深度移动端重设计。
- **环境**：本 feature 在本地开发环境验证即视为完成，公网部署 / HTTPS / 域名 / 数据隔离 / 限流 / 安全加固**全部不在本 feature 范围**，留待下一发版。
- **依赖**：DeepSeek API Key 在 `.env` 中已配置且额度充足，本地开发期间能稳定响应。
- **不引入新后端依赖**：本 feature 不引入 slowapi、APScheduler 等新中间件；可能新增的后端代码限于"写类工具的待确认包装层"和（可选的）"角色快速切换 API"。
- **测试边界**：本 feature 不要求自动化 E2E 测试覆盖移动端真机；以 PC 浏览器开发者工具的移动视口模拟为最低验收手段。
