## 三个值得想清楚的问题

### 一、PRD会消亡吗？

**杨老师的判断：** 随着业务上下文、系统上下文、源代码上下文越来越丰富，PRD的价值会持续衰减。未来产品经理的核心工作可能是：提供业务上下文，明确设计约束、规则、强逻辑——由此产生的交互和代码，AI来生成。

我同意这个方向，但想补充一个前提：**PRD消亡的条件是"上下文足够丰富且可信"**。

PRD的本质是人与人之间的沟通媒介——产品经理把业务需求翻译成开发能理解的语言。当 AI 能直接理解业务上下文、直接读懂系统现状、直接从约束条件生成方案，这个翻译层的必要性就在衰减。

但"上下文足够丰富"在 ToB 领域需要漫长的积累。企业有大量遗留系统、非结构化的业务流程、隐性的组织知识——这些东西数字化程度低，AI 短期内很难自动理解。所以我的判断是：**PRD不会消亡，而是转型**。

传统 PRD（面向人的沟通文档）会逐渐被"面向 AI 的上下文描述"替代——更结构化、更机器可读、更关注约束和规则而非界面描述。产品经理的工作重心从"写文档"变成"定义上下文和约束边界"。

与此同时，杨老师提到需要一个"AI 技术架构师"角色负责技术部分，与 PM 协同。这个角色现在还没有标准定义，如何协作也在探索中。但这个组织形态的变化，本身就是值得持续观察的命题。

有一点是确定的：在相当长的时间里，传统系统太多、积累的 PRD 规范太深，ToB 领域的传统产研模式不会快速消失。所以我们现在做的这个 `check-prd`，既是对当下实践的改进，也是对这个转型过程的一次提前预演。

---

### 二、把最佳实践写成 Skill，会不会被 LLM 偷师学走？

**杨老师的担忧：** 企业把自己的最佳实践整理成 Skill（文本化的规范），持续喂给 AI 使用。但 LLM 一旦升级，这些最佳实践会不会被吸收进通用模型，最终变成人人都有的能力？辛苦建立的护城河，反而加速了自己的竞争优势被稀释？

这个问题需要拆成两层来看。

**第一层：Skill 会被训练进去吗？**

技术上，Skills 是推理时的提示词（Prompt），不是训练数据。你在对话框里输入的内容、调用 Skill 时加载的文件，不会直接成为模型的训练素材——至少在正常的 API 使用场景下如此。所以"直接偷师"这条路基本不成立。

**第二层：更本质的风险是什么？**

真正的风险不是"被偷走"，而是**"被超越"**。

LLM 能力在持续升级。今天你手写 Skill 才能做到的分析深度，明年可能是模型的基础能力。这不是偷——是整个行业的知识水位在涨。你的 Skill 会从"竞争优势"变成"行业基准"。

但这里有一个反直觉的结论：**这个趋势不会让 Skill 失去价值，只会让 Skill 的迭代压力变大**。

因为 LLM 的通用能力越强，专业深度反而越难被替代。当所有人都能用 AI 做基础 PRD 检查，真正有价值的是"在这个行业、这个产品类型、这个阶段，最关键的判断是什么"——这种判断来自真实的业务经验，不是通用知识。

杨老师的情况更特殊：他的 Skills 来自他自己写的书，那些书本身就是公开出版的知识。所以这里不存在"被偷"，只有"被加速传播"。这反而是好事——Skills 开放出去，用的人越多，反馈越多，迭代越快，最终受益的还是方法论本身。

## check-prd：基于两本书生成的 PRD 检查 Skill

杨老师是《决胜B端（第二版）》《决胜体验设计》的作者，两本书系统讲了 B 端产品从业务调研到架构设计到交互设计的完整方法论。`check-prd` 就是把这两本书加上《决胜B端PRD模板 v2.0》，提炼成 AI 可以直接执行的检查规范——一个 Claude Code Skill。

直接问 AI「帮我检查这份PRD」，通常得到的是：「文档整体结构完整，建议补充用户场景描述，加强异常处理设计。」这种话对哪份 PRD 都成立，说了等于没说。

问题不是 AI 不懂 B 端，是没有给它一个**足够明确的检查框架**。产品经理评审 PRD 时，脑子里有体系——业务逻辑是否成立、数据模型是否合理、角色权限是否考虑——每个判断背后都有对应的专业标准。书里的方法论，就是这套标准的来源。

### 一票否决机制：8 个重大风险项

14 个维度之外，有 8 个问题属于一票否决——任何一个触发，建议暂停推进，优先修复：

| 编号 | 风险项 | 触发条件 | 适用范围 |
|---|---|---|---|
| R1 | 产品定位缺失 | 读完 PRD 无法一句话说清为谁解决什么问题 | 全部 |
| R2 | 核心业务流程缺失 | 无法从 PRD 还原出业务从开始到结束的完整路径 | 全部 |
| R3 | 数据模型严重缺陷 | 涉及多业务实体但无 ER 图，或 ER 图关系明显错误 | 业务型/交易型 |
| R4 | 角色权限完全缺失 | 涉及多角色但完全没有权限设计 | 全部 |
| R5 | 功能存在致命断裂 | 核心业务流程中存在功能断点，业务无法走通 | 全部 |
| R6 | 需求与业务严重脱节 | PRD 设计的功能与实际业务需求不符 | 全部 |
| R7 | 合规性风险 | 强监管行业（金融/医疗/教育）未考虑任何合规要求 | 涉及强监管行业 |
| R8 | SaaS 无多租户概念 | 商业化 SaaS 产品无租户隔离设计 | 商业化 SaaS |

---

所有发现按 P0（核心流程/合规风险）→ P1（上线前必修）→ P2（有余力再做）→ P3（锦上添花）四级输出。

## 如何使用

### 方式一：Claude Code Skill（推荐）

```bash
# Mac/Linux
git clone https://github.com/pmyangkun/check-prd-skill.git && cd check-prd-skill && bash install.sh
# Windows：同上，最后换成 .\install.ps1
```

安装后两行命令开始使用：
```
/model claude-opus-4-6      # 切换 Opus（深度分析必需）
/check-prd 你的PRD文件.pdf
```

不熟悉 git？直接下载 ZIP 解压运行安装脚本：`github.com/pmyangkun/check-prd-skill`

### 方式二：通用 Prompt（ChatGPT / Gemini / 其他 LLM）

仓库里有合并好的单文件 `check-prd-universal-prompt.md`，约 150KB。全选复制，新建对话，先粘贴 Prompt 内容，再粘贴 PRD 内容，一起发送。

注意：通用 Prompt 方式分析质量低于 Claude Code Skill 版本，因为所有内容一次性加载，上下文压力更大。建议使用支持长上下文的模型（GPT-4o、Gemini 1.5 Pro、Claude 等）。

## 真实验证：8 份 PRD 的检查结果

这次找来了 8 份企业真实 PRD，产品类型从 CRM 底层设计到小功能迭代都有，来自不同团队和作者。全部跑完整检查流程。以下选取两个典型案例展示，均已匿名处理。

---

### 案例一：6 个重大风险项——某金融知识库 MRD

**背景：** 某金融服务平台的知识库产品 MRD，约 20 页，用于管理和分发金融知识内容（贷款、投资相关）。产品定型：定位为工具型内容产品，商业化运营。

**触发风险项：R1 产品定位缺失 / R3 数据模型缺失 / R4 角色权限缺失 / R6 需求与业务脱节 / R7 金融合规缺失 / 跨端一致性风险**

本次 8 份 PRD 中触发风险项最多的一份。典型发现：

> **发现：产品实质上是业务型系统，但按工具型设计** ｜ P0
>
> 文档涉及「知识条目—标签—分类—版本—审核状态」等多个有关联关系的业务对象，但按工具型产品处理，缺少 ER 图和状态机。工具型产品可以不画 ER 图，但这个产品的复杂度已经是业务型。
>
> **改进示例：** 在定位章节明确声明「本产品为业务型知识管理系统，核心业务对象：知识条目、标签体系、审核流程」，并补充 ER 图和状态机（至少覆盖：草稿 → 待审 → 已发布 → 已下线 四个状态及逆向流转）。

> **发现：涉及金融内容，全文无任何合规声明** ｜ P0
>
> 金融知识内容（贷款、投资类）存在误导用户的法律风险。平台方需要明确内容审核标准、免责声明机制、监管合规要求。这不是锦上添花，是法务问题。
>
> **改进示例：** 第 3 节「内容规范」下增加「合规要求」子节：① 所有金融内容需经持牌人员审核；② 页面底部必须展示固定免责声明文案；③ 敏感内容（收益承诺、风险评级）需标注并触发二次确认。

**核心问题：** 用写工具产品的方式写了一个业务型产品。这个错位导致架构、数据、合规层面出现系统性缺失，而不是单点漏洞。

---

### 案例二：组件级 UI 解剖——某 CRM 客户管理模块 PRD

**背景：** 某公司 CRM 系统的客户管理模块迭代 PRD，覆盖客户列表、客户详情、客户分级申请等功能，约 30 页，有原型图附件。产品定型：业务型管理软件，自研内部系统。

**触发风险项：R1 产品定位缺失 / R3 ER 图缺失 / R4 角色权限缺失**

这份 PRD 的常规风险项问题不赘述。值得单独展示的是维度09（交互设计）的组件级分析——**《决胜体验设计》方法论直接落地的部分**，对最终体验影响大，但最容易被忽视。

> **发现：38 个字段的分级申请表单无分组、无暂存** ｜ P0
>
> 客户分级申请表单平铺 38 个字段（基本/联系/业务/财务信息），无分组标题，无进度指示，无暂存。超过 10 个字段用户中途退出概率急剧上升；无暂存意味着接个电话就得从头再来。
>
> **改进示例：** 拆成 4 步分步表单（每步自动暂存），或保持单页但加分组标题 + 右上角「保存草稿」按钮。

> **发现：「分配客户」用了弹窗，但场景需要抽屉** ｜ P1
>
> 弹窗遮挡客户信息，但分配时恰恰需要对照客户的城市、行业、历史跟进情况来决定分给谁——信息和操作无法同时进行。
>
> **改进示例（来自《决胜体验设计》弹窗选型原则）：** 改用右侧抽屉，客户信息保持可见。原则：弹窗适合「确认」类操作，抽屉适合「需要参照上下文」的操作（分配/转移/审批）。

**核心问题：** 这份 PRD 在业务逻辑层面没有大问题，但交互设计层面存在大量「习惯性选择」——选了最顺手的控件，而不是最合适的控件。这些问题单独看不严重，叠加起来会让整个系统的使用体验比它本来应该达到的水平低一个档次。

## 数字总结

| 指标 | 数据 |
|---|---|
| 平均触发重大风险项 | **3.1 个**（最少 1 个，最多 6 个） |
| 角色权限矩阵（R4）缺失率 | **8/8，100%** |
| 产品定位缺失（R1） | 6/8，75% |
| ER 图缺失（R3，适用产品中） | 100% |
| 数据合规问题（R7） | 4/8，50% |

**角色权限矩阵缺失率 100%** 是这次验证最令我意外的数字。

这是最基础的设计要素之一——谁能看什么、谁能做什么。《决胜B端》里专门有一章，PRD 模板里有标准格式，结果 8 份真实企业 PRD 没有任何一份完整给出这个表格。

原因不是 PM 不知道权限重要，是大家都默认「开发懂的，不用写」。这个假设最终会变成上线前的临时补丁、争论和返工。

## 后记

`check-prd` 目前可以免费取用：`github.com/pmyangkun/check-prd-skill`

回到开头那两个问题：PRD 会转型但不会消亡；Skill 的价值不在于独占知识，而在于持续迭代判断力。

## 下期预告

`check-prd` 的 14 个维度定义了"一份完整的需求应该包含什么"。下一集，我们要把这件事**反过来用**。

既然工具知道"好需求应该有哪些要素"，那它就可以反向提问——引导杨老师把 SFA CRM 的业务上下文、需求上下文、企业架构上下文，一项一项补充完整。

这个上下文初始化完成后，就是 Spec Coding 设计阶段的起点：不写 PRD，直接基于上下文，让 AI 参与完成系统设计。

具体来说，下一集会覆盖：

- **业务上下文**：SFA CRM 要服务的销售团队是什么样的组织结构？业务流程是什么？痛点在哪里？
- **需求上下文**：系统要支撑哪些核心场景？约束条件和强逻辑有哪些？
- **企业架构上下文**：这个系统在整个技术栈中的位置？与哪些系统集成？
- **设计阶段输出**：基于以上上下文，AI 辅助完成系统边界划定、核心对象建模、模块划分

这是真人秀真正开始的地方。敬请关注。
