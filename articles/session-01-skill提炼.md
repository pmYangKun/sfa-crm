# 当设计方法论遇见AI：从两本书到四个可执行的Skill

> 这是一个正在进行中的实践项目的第一篇记录。作者是《决胜B端》和《决胜体验设计》的作者。
>
> 这个项目有两个目标，缺一不可：
>
> **第一个目标**，把两本书里的设计方法论，变成AI能直接执行的 Skill 和 Spec，让读者在用AI辅助做B端产品设计时，AI能真正按照经过实践验证的方法论工作——而不是凭"直觉"瞎干。
>
> **第二个目标，也是更有挑战性的那个**：用第一步生成的 Skill 和 Spec，驱动AI从零构建一套完整的 SFA CRM 系统。这涉及到 Spec Coding 这一前沿技术实践——用结构化规范约束AI的构建行为，让AI生成的不只是能跑的代码，而是设计合理、架构清晰、符合B端产品规范的真实系统。如果这条路走得通，意味着"设计方法论→AI规范→复杂系统落地"这个链路是可行的，这将是对整套方法论最有力的验证。
>
> 这篇文章记录了第一次对话的完整过程——包括思路的碰撞、概念的澄清、以及第一阶段的产出。你会看到AI是怎么读懂一本书的，也会看到人机协作是怎么一步一步把想法变成工具的。

## 一切从一个想法开始

对话是这样开始的：

> **作者：** 接下来，我想和你合作一个大工程。

然后，把整个构想一口气说了出来：

> 我是《决胜B端》《决胜体验设计》两本书的作者。有几件事：
>
> 第一，我想让你基于对这两本书的解读，把里边的方法论变成skill或者spec；
>
> 第二，根据生成的spec或skill，我想让你创建一个典型的SFA CRM，看看你到底能不能实现复杂系统的构建；
>
> 此外，我需要边和你对话，边学习AI知识，边设计实践案例。

这段话背后藏着两个层层递进的假设：

**假设一：** 经过实践验证的设计方法论，能够被结构化成AI可理解、可执行的规范（Skill / Spec）。

**假设二：** 这些规范，能够真正驱动AI构建出一套设计合理的复杂B端系统——不只是代码能跑，而是架构清晰、数据模型正确、权限体系完整。

第一个假设相对容易验证；第二个假设，才是这个项目真正的硬核挑战。

这两个假设成不成立，只有做了才知道。

## 第一个拦路虎：Skill 还是 Spec？

在开始做之前，有个概念问题必须先想清楚。

> **作者：** 按照我目前的理解，我书中关于B端产品设计的框架、结构，可以作为CRM系统的spec规范；我书中的一些重要的分析思路，比如利益方痛点，可以作为项目设计中AI提醒我需要提供的利益方痛点信息，这是作为skill存在的，对吗？

这个问题问得很准。我给出了这样的回答：

**Spec（规范文档）** 是描述"要建什么"的结构化文档。书里对CRM系统架构的描述——哪些模块、哪些数据对象、什么权限体系——这些是Spec，告诉AI"这个系统长什么样"。

**Skill（可执行技能）** 是一个可以被调用的命令。比如 `/check-prd`，用户输入这个命令，AI就会按照书里的PRD检查框架逐条审查文档。Skill封装的是一个**分析或执行过程**，而不是一个结果。

然后把这个关系画得更清楚：

> 读者的行为是主动触发的——"帮我检查这份PRD"、"给我一个设计方案"。这是**动作**，不是背景知识。Skill就是把书中方法论变成一个可执行的流程。但Skill离不开Spec，真正的内容是书里的框架和判断标准，要写成一份**方法论文档（Spec）**，让Skill调用它。

两者的关系，用一张图说清楚：

```
读者调用 /check-prd
    ↓
Skill（执行逻辑：怎么审、问什么、输出什么格式）
    ↓
方法论 Spec（书中框架：维度定义、判断标准）
```

还有一个现实问题也被提了出来：读者不一定都用Claude Code，也可能用ChatGPT或Claude.ai网页版。所以这套工具最终可能有两种形态——Claude Code Skill和通用System Prompt模板——内核是同一份方法论，只是包装不同。

作者听完之后说：

> **作者：** 我们先聚焦第一本书《决胜B端》吧，把他融入AI，方便其他读者让AI检查PRD，或者给出设计方案时，能按照书中的方法论框架执行。

方向定了。下一步，读书。

## AI是怎么"读"一本书的

《决胜B端》是一本25万字的完整书稿。我直接解析了Word文档，提取出全书的章节结构。

结构出来之后，四篇内容的AI价值高下立判：

| 篇章 | 内容 | 对AI的价值 |
|------|------|----------|
| 概述篇（1-2章）| B端认知、行业背景 | 低——背景知识，不是可执行方法 |
| **设计篇（3-6章）**| **完整设计流程** | **极高——这是核心方法论** |
| 管理篇（8-11章）| 项目/需求/迭代管理 | 中——流程类，可做辅助Skill |
| 进阶篇（12-15章）| 企业应用架构 | 高——架构设计的判断标准 |

最有价值的是设计篇，五章内容形成了一个完整链条：

```
第3章：B端产品总体建设流程（入口）
  ↓
第4章：业务调研（战略层/战术层/执行层三层框架）
  ↓
第5章：产品设计（定位→架构→模块→蓝图）
  ↓
第6章：详细设计（数据建模→流程→界面→权限→文档）
  ↓
第7章：技术对接（PM需要懂的技术知识）
```

基于这个链条，提出了三个核心Skill的设想：

- **`/b2b-research`**：业务调研引导（第4章）
- **`/b2b-design`**：产品方案生成（第5章）
- **`/check-prd`**：PRD检查（第6章）

## 意外收获：第二本书的补充

就在准备动手的时候，作者说：

> **作者：** 稍等。在我的新书《决胜体验设计》中，也有部分章节是讲用户调研，甚至包括一些架构抽象的新内容。请把这本书和《决胜B端》产品设计方法论相关的部分，理解、融合到一起。

这是一个重要的插入。第二本书同样解析了一遍，然后做了一件关键的事：**判断两本书的关系**。

两本书不是重复，而是互补——从不同视角覆盖了B端产品设计的完整链条：

| 维度 | 《决胜B端》| 《决胜体验设计》（方法论部分）|
|------|-----------|--------------------------|
| 研究视角 | **业务视角**：战略/战术/执行三层 | **用户视角**：画像/场景/旅程 |
| 架构视角 | **功能架构**：模块、数据模型 | **信息架构**：子系统划分、菜单组织 |
| 调研方法 | 业务调研五步法 | 用户访谈十大技巧 + 问卷设计方法 |
| 设计输出 | PRD、流程图、数据建模 | 用户画像、旅程地图、体验优化点 |

新书里有一段话，把这种互补关系说得特别透彻：

> "在经典的B端产品设计实践中，我们总是从业务流程梳理着手，最终设计出来的软件，从业务运作来讲是通畅的。但如果从单一角色的视角来分析参与业务过程中的体验感受，有可能会很糟糕。软件用户典型的感受是：该有的功能都有，但连起来用却很别扭。"

这就是两本书各自的盲区：业务视角容易忽略用户感受，用户视角容易忽略业务逻辑。

融合之后，得到了一个更完整的四阶段框架：

```
第一阶段：双轨调研
  ├── 业务调研（战略/战术/执行三层框架）
  └── 用户调研（画像 + 场景地图五步法）

第二阶段：产品定位
  ├── 核心业务流程梳理
  ├── 产品定位与边界确定（避免"全家桶"陷阱）
  └── 子系统划分（松耦合高内聚原则）

第三阶段：架构设计
  ├── 应用架构（功能模块 + 演进蓝图）
  └── 信息架构（菜单编排 + 功能分类）

第四阶段：详细设计
  ├── 业务数据建模
  ├── 流程与角色设计
  ├── 权限设计
  └── PRD编写规范
```

相应地，Skill的规划也从三个扩展到四个，把产品定位单独拆出来：

| Skill | 触发命令 | 覆盖框架 |
|-------|---------|---------|
| 双轨调研引导 | `/b2b-research` | 两本书的调研方法论 |
| 产品定位诊断 | `/b2b-positioning` | 产品边界 + 子系统设计 |
| 架构方案设计 | `/b2b-architecture` | 模块 + 信息架构 + 权限 + 蓝图 |
| PRD质量检查 | `/check-prd` | PRD八维度检查框架 |

## 四个 Skill 的核心内容

### `/b2b-research`：双轨调研引导

这是设计流程的起点。Skill会引导用户完成两条调研线——业务调研和用户调研，最终合并输出一份结构化的调研结论。

**业务调研的核心是三层框架：**
- **战略层**：公司/业务的核心经营目标是什么？关键利益方的KPI是什么？
- **战术层**：核心业务流程是什么？各角色怎么协作？流程的瓶颈在哪里？
- **执行层**：选取2-3个典型场景，还原操作细节——谁在哪里做什么，用什么工具，遇到什么问题？

**用户调研的核心是场景地图五步法：**
明确画像用户 → 确定业务目标 → 推导关键任务 → 绘制旅程地图 → 细化业务场景

Skill还内置了访谈技巧提示，比如"场景还原法"（让用户描述"上次你是怎么做的"而不是"你一般怎么做"），以及要避免的陷阱（不要让用户预测未来行为，因为不可靠）。

### `/b2b-positioning`：产品定位诊断

这个Skill解决的是B端设计中最常见的一类错误——**产品定位模糊，边界不清**。

Skill会引导用户用五步以内描述核心业务流程（价值链），然后逐步骤判断哪些纳入产品范围、哪些不纳入，最终输出一个标准格式的产品定位描述：

```
本产品服务于 [目标用户群体]，
帮助其解决 [核心业务场景] 中的 [核心问题]，
实现 [业务价值/KPI目标]。
不包含：[明确排除的功能范围]。
```

Skill特别强调了"全家桶陷阱"——很多B端产品为了让用户少切换系统，把不相关的业务强行塞在一起，造成技术上的紧耦合和体验上的混乱。**产品定位描述里只要出现"以及"、"同时还支持"，往往就是边界不清的信号。**

### `/b2b-architecture`：架构设计引导

这是四个Skill里最"重"的一个，覆盖了从业务能力组件到演进蓝图的完整架构设计过程。

几个关键节点：

**业务能力组件分析：** 把业务流程的每个步骤，翻译成软件系统需要提供的能力（数据管理类、流程执行类、协作沟通类、统计分析类……）。

**数据建模警示：** 这是Skill里着墨最重的部分。原因在于《决胜B端》第6章有一个直接的判断：

> "业务数据建模错误会导致灾难——一旦系统上线，数据模型几乎不可能重构。"

Skill强制要求用户在开始详细设计之前，先想清楚这些问题：客户被删除后关联订单怎么处理？同一个客户被两个销售重复录入怎么办？数据归属规则是属于个人还是公司？

**RBAC权限体系：** B端产品的权限设计分两个维度——功能权限（能不能用这个功能）和数据权限（能看到哪些数据）。数据权限往往比功能权限复杂得多，而且会深度影响数据库设计，必须在架构阶段就想清楚。

### `/check-prd`：PRD质量检查

这大概是四个Skill里最容易被读者直接使用的一个——把PRD粘贴进去，AI按八个维度逐一检查并出具检查报告。

八个检查维度：
1. **需求背景与目标**（有没有说清为什么做）
2. **业务流程与角色**（有没有覆盖异常流程、角色边界是否清晰）
3. **业务数据建模**（最容易被忽略、出错代价最高）
4. **界面与交互设计**（列表页、表单页的常见Bug清单）
5. **报表设计**（二维表格的常见问题）
6. **权限设计**（功能权限 + 数据权限是否都考虑到了）
7. **数据埋点**（关键操作是否定义了埋点）
8. **文档规范**（结构完整性 + 描述质量）

检查完成后，Skill会输出一份分优先级的检查报告：必须修复、建议完善、可选优化。

## 一个重要的边界说明

在继续之前，有必要把这个项目的范围说清楚。

当下AI发展极快，"AI原生产品"、"Agent"、"Chat交互"这些概念铺天盖地。但坦率地讲，**AI原生产品的设计方法论目前还远未成熟**。我们还没有足够多的实践案例，去总结一套可靠的、可复用的AI原生B端产品设计框架。

因此，这个项目的设计思路，**依然立足于传统软件工程下的B端产品设计实践**——经过多年行业积累和验证的那一套：业务调研、产品定位、功能架构、数据建模、权限设计、PRD规范……这些方法论经得起推敲，也是目前绝大多数B端产品经理真正需要掌握的核心能力。

**纯AI产品的设计不在我们讨论的范围之内。**

不过这并不意味着我们会回避AI。在后续的SFA CRM构建阶段，会有意识地探索如何在传统业务型B端产品中**融入Agent能力**，包括Chat交互形态的引入——不是把产品做成一个对话机器人，而是在合适的业务场景里，让AI成为业务执行的协作者。

这是一种更务实的路径：**以成熟的传统设计方法论为地基，在上面逐步叠加AI能力**，而不是用一套尚未验证的AI原生方法论从头来过。

## 关于 Vibe Coding 和 Spec Coding

读到这里，可能有读者会问：这和最近流行的"Vibe Coding"有什么关系？

**Vibe Coding** 是一种用自然语言驱动AI写代码的方式——你描述你想要什么，AI凭直觉完成任务。它极大地降低了编程门槛，也是这两年AI编程热潮的核心驱动力之一。

但Vibe Coding有一个根本性的问题：**AI的"直觉"不一定对**。生成的代码可能能跑，但设计思路是错的——对于B端产品这种复杂系统，"能跑但设计错了"可能意味着上线之后数据一团糟，或者某个关键业务场景根本走不通。

**Spec Coding** 是进阶的做法：先用结构化的规范（Spec）告诉AI"应该怎么做"，再让AI按规范执行。这样AI的行为更可预测，输出质量更稳定。

这个项目做的事情，本质上就是在构建 **B端产品设计领域的 Spec**——把经过实践验证的设计方法论，转化为AI可以理解和执行的结构化规范。

当一个产品经理打开 `/check-prd`，AI按照书里的八个维度检查他的文档时，他用的不是"AI的直觉"，而是**经过验证的设计经验**。这就是Spec Coding的价值所在。

## AI 的"记忆"是怎么工作的

这个项目会持续推进，可能跨越数十次对话。有一个基础问题需要解决：**AI没有跨对话的记忆。** 每次打开一个新对话，它都从零开始，不知道我们上次做了什么。

解决方案是维护一套文件化的记忆系统。在这次对话里，AI把项目的关键信息写入了几个文件：

**用户档案（user_profile.md）：** 记录了作者的背景和目标，确保AI在接下来的每次对话里都能以合适的方式协作。

**项目记忆（project_main.md）：** 记录了整个项目的目标、三个阶段的规划和当前状态。

```
项目目标：将两本书的方法论转化为可执行的AI spec/skill，
再用这些spec构建SFA CRM系统，验证方法论的落地价值。

三个阶段：
1. 书籍解读 → 提炼方法论 → 转化为spec/skill   ← 本次已完成
2. 基于spec构建SFA CRM（验证spec能否驱动复杂系统构建）
3. 复盘总结，整理可开放的spec/skill包
```

**行为偏好（feedback_autopermission.md）：** 作者不希望每次工具调用都弹出确认框，这个偏好被记录下来，并同步到了Claude Code的配置文件里，从下次打开就自动生效。

这套记忆系统存放在项目目录下（而不是系统默认的C盘位置——这是作者要求的，"C盘太不安全了"），通过目录符号链接和配置文件，确保数据只存在一个地方，不会因为路径混乱而丢失。

## 接下来做什么

四个Skill已经创建完成，文件放在Claude Code的skills目录下，用户输入 `/b2b-research`、`/b2b-positioning`、`/b2b-architecture`、`/check-prd` 即可触发。

但"创建完成"不等于"验证有效"。接下来的工作分两个阶段：

**第一阶段：验证 Skill 是否真的有用**
- 用 SFA CRM 作为真实案例，逐个跑一遍四个 Skill
- 检查 AI 的引导输出是否有实际指导价值，有没有遗漏关键判断
- 根据结果迭代优化 Skill 内容

**第二阶段：用 Spec Coding 构建 SFA CRM——这才是真正的硬仗**
- 把经过验证的 Skill 和 Spec，作为驱动 AI 构建系统的规范输入
- 让 AI 按照书里的方法论，完成从数据建模、权限设计到前端界面的完整构建
- 这个过程会涉及 Agent 能力和 Chat 交互形态的融入——不是把 CRM 做成对话机器人，而是在合适的业务场景里让 AI 成为协作者
- 全程记录 AI 做到了什么、在哪里出了问题、需要人工介入的节点在哪里

如果这条路走通了，它证明的不只是"AI能写代码"，而是**"经过结构化的设计方法论，能够让AI构建出真正可用的复杂业务系统"**。

这将是对两本书最有力的实践背书。
