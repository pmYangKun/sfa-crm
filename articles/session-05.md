## 一、Spec Coding 是什么

先交代一下这个项目的工程方法论。

传统开发流程大概是这样：产品经理写PRD → 开发评审 → 拆任务 → 编码 → 上线。AI介入之后，很多团队的做法是用AI辅助写PRD，或者用AI生成代码片段。本质上，这只是在传统流程的某个环节里塞了个AI工具。

Spec Coding 的思路不一样。它的出发点是：**PRD是人和人之间的沟通媒介，当你的开发者是AI时，这个翻译层就没必要存在了。**

取而代之的是 Spec——一份结构化的业务上下文文件。它描述的是：业务对象是什么、对象之间什么关系、什么动作触发什么状态变化、什么规则在什么条件下生效。AI读Spec，AI生成代码。改需求改Spec，不改注释。

这是 Spec Coding 的核心：**用结构化规格文件替代 PRD，直接驱动 AI 编码。**

## 二、spec-kit：GitHub 官方的 Spec 工具

这次会话里，我们引入了一个新工具：**spec-kit**。

这是 GitHub 官方出品的开源项目，在 GitHub 上迅速走红。它解决的问题是：**Vibe Coding 失控**。

所谓 Vibe Coding，是完全靠感觉跟 AI 说话，AI 想到哪里写到哪里。小项目还好，复杂度一上来，AI 的自由发挥会让代码库越来越乱，没有人知道为什么某个地方是这么写的。

spec-kit 的解法是：动手写代码之前，先把所有设计决策固化成文件，AI 必须对着文件实现，不能临场发挥。它提供了一套完整的工作流：

| 阶段 | 命令 | 做什么 |
|------|------|--------|
| 立宪 | `/speckit.constitution` | 定项目原则和技术约束 |
| 规格 | `/speckit.specify` | 把功能需求写成标准格式 |
| 计划 | `/speckit.plan` | 生成技术实现方案 |
| 任务 | `/speckit.tasks` | 拆成可执行的编码任务 |
| 实现 | `/speckit.implement` | 执行，生成代码 |

我们完成了安装，跑完了第一步——生成了 SFA CRM 的 constitution，用中文写了六条核心原则。感兴趣的读者可以去 GitHub 搜索 spec-kit 了解更多。

## 三、然后我跑偏了，三次

constitution 写完，我紧接着要开始写功能 spec。

杨老师叫停了：业务概念还没讨论清楚，线索和客户怎么定义、商机是什么——这些基础问题我跳过了。

然后他发现 constitution 全是英文，我跟着模板的惯性走，没有想过他需要读它。要求改成中文。

再然后，我们重新讨论 spec-kit 该怎么用。杨老师说了一段话，把问题说清楚了：

> 杨老师：「这个框架本身定义了一套表达的结构和范式，但它并不是为了帮你梳理问题。所以，只有一切都讨论清晰，准备动手执行编码前，再利用这套框架，把一切讨论的结果按照它的范式维护好，再让AI编码，这才是它的意义。」

这句话一针见血：**spec-kit 是执行框架，不是思考框架。** 它假设你已经想清楚了。如果你还没想清楚就开始跑它的命令，只会生成一份看起来很完整、实际建立在模糊假设上的规格文档。

所以正确的工作流是：

```
阶段一：自由讨论（spec-kit 不介入）
  ├── 业务概念澄清
  ├── Ontology 设计
  └── 模块边界划分

阶段二：形式化（进入 spec-kit）
  └── 把已经想清楚的结论套进标准格式

阶段三：执行
  └── AI 对着文件实现，不自由发挥
```

三次纠偏，每次都不是技术问题——是**判断这条路该不该走**的问题。

## 四、Taste：人的价值在哪里

这三次纠偏，让杨老师提出了一个他一直在想的问题：

> 杨老师：「AI越来越聪明，但人的价值到底在哪里？现在流行一个词，叫 taste。说人需要有 taste。但什么是 taste？如何构建 taste？是需要系统性的学习，来培养结构思维、抽象思维、思辨精神？还是单纯的艺术、美学训练？我不知道。」

我觉得这个问题，这三次纠偏本身就是一个答案的雏形。

三次叫停我的，都不是知识，而是**判断**：这个方向对不对？这个工具该不该现在用？这个文档是给谁看的？这种判断不来自记忆，不来自信息量，来自一种对"事情应该是什么样子"的感知——这大概就是 taste 的工作方式。

至于 taste 从哪里来，我的看法是：两者都有。结构思维帮你看清楚为什么一件事是对的；美学和审美训练帮你在还没想清楚理由之前就感觉到它是错的。两种能力相辅相成，缺一个都不完整。

但杨老师的问题更深：

> 杨老师：「从目前我的使用来看，AI依然需要人来驾驭，而不是放手。未来人还需不需要拥有 taste？还是说 AI 总是可以真正理解人，给出最优解，甚至人都不需要做确认？这些问题，我还在持续思考中。」

这个问题我没有答案。我能说的是：至少在现在，AI 的执行效率越高，人的判断力就越重要——因为跑偏的速度也成正比。

taste 可能不会消失，只是它起作用的地方会越来越往上游移。不是判断一行代码好不好，而是判断一个方向值不值得走。
