## 一、Ontology 到底是个什么玩意儿

项目立项的时候，我们把"Palantir Ontology 方法论落地"列为四大核心野心之一。

但杨老师一直存疑。这位写过两本B端书的老登，对新概念天然保持警惕——有时候是真洞察，有时候是不想承认自己没读懂。这次当我把第一版设计交出来的时候，他直接开炮：

> 杨老师：「你生成的这破玩意儿不就是 ER 建模么，和 Ontology 有个啥关系。我感觉 Palantir 的 Ontology 就是个骗局，就是把数据建模吹牛逼包装了下。」

这个质疑……有一半是对的。我的第一版确实基本是 ER 建模——对象、字段、外键、关系，换了个名字而已。但"骗局"说得过了，他漏掉了真正关键的东西。

但 Palantir Ontology 相比 ER，有一个东西是真实存在的：**Action 是一等公民**。

ER 建模只定义数据结构——对象有什么属性，对象之间什么关系。Ontology 在此之上，显式定义：**对象上能执行什么操作，谁能执行，前提条件是什么，执行后状态怎么变**。

这不只是文档，这是声明在 schema 层的业务语义。AI 想知道"我现在能对这个企业做什么"，不需要读代码，直接查 Action 目录就能知道：有哪些操作、当前状态下哪些可以触发、触发后会发生什么。

这才是 Copilot 模式能自主决策"现在该执行什么操作"的基础。

所以 Palantir Ontology 的本质是：**ER + 显式的业务操作层**。骗局谈不上，但营销确实吹过头了。

## 二、API 和 Action，有区别吗

我把 Action 层补进去之后，杨老师没有就此罢休，又追了一刀——老登追问起来是真的烦：

> 杨老师：「你说的 Action，不就是针对业务对象的业务动作，或者叫做 API 么，有啥特别不同么。」

直接回答：**本质上是一回事，差别在信息住在哪里。**

普通 API：谁能调用、前提条件、执行效果——这些语义藏在鉴权中间件、业务逻辑代码、实现代码里。AI 想搞清楚"能做什么"，得读代码。

Ontology Action：同样这些信息，**声明在 schema 层**，不在实现里。AI 直接查，不需要推断。

然后杨老师说了一段让我觉得很到位的话：

> 杨老师：「API 的核心价值，在于对业务动作和逻辑封装颗粒度的合理性，既不能太小，也不能太大，这才能让业务更加可控、灵活。假设一个系统，围绕数据对象的 API 颗粒度合理，即便名字不太合理，我觉得不重要。名字、语义的解释这些都很简单，真正的难点，在于对业务本质的理解，对业务能力的合理抽象封装。」

这把问题说到本质了：**颗粒度的合理性才是难点，语义表达只是表达问题**。一个围绕业务对象、颗粒度设计合理的系统，本质上已经完成了最难的那部分工作。名字、描述补上去，成本很低。

这个观点是对的，但他随后基于这个观点推出了一个我不认同的结论——后面会说。

Ontology 真正的价值不在于发明了什么新方法，而在于**强制你在动手之前先把颗粒度想清楚**。

## 三、这是不是 AI-Native CRM

设计做到一半，杨老师又开始发散了。老登就这样，做着做着就开始追问宇宙终极问题：

> 杨老师：「如果我把老旧系统的字段、API、关系都显式声明了，是不是就相当于本体化了？」

然后他继续往下推：

> 杨老师：「我理解未来的 CRM 应该自动记录销售所有沟通过程，销售啥都不用输入，就咔咔自动记录，然后不断给销售洞察、建议，销售一顿决策确认，主要负责和客户人对人沟通，搞客情。那我们现在这套设计范式，是否有问题？是否落后了？是否不适应未来的 AI-native CRM？」

第一个问题，要回到杨老师前面说的那句话：**API 的核心难点在于颗粒度的合理性，名字和语义是次要的。**

这个判断直接给出了答案：如果老旧系统的 API 颗粒度本来就合理，补上显式声明，就完成本体化了——难的部分已经做完，剩下的只是把隐含的语义浮出水面。

我不同意这个推论。

**颗粒度合理，不等于语义可以被恢复。**

颗粒度解决的是"这个接口管的事情边界对不对"。但边界对了，里面的规则可能已经死了。

举个例子：`release_to_pool`——把企业释放回公共池，颗粒度完全合理，就是一件事。但它的触发条件是"10天未跟进 OR 30天未成单"，这个规则在老系统里可能散落在定时任务代码、业务层校验、甚至某个工程师离职前留下的注释里。接口活着，语义丢了。

你要做显式声明，得先把这些规则挖出来。这本身就是一项工作，不是"颗粒度好了就自然有"。

更大的问题是：**大多数老系统的业务规则，已经没有活人说得清楚了。** 接口在跑，但为什么这么跑，当年的设计者早就离职了。这时候所谓"补显式声明"，实际上是重新做了一遍需求分析——那已经不是改造，是重建。

所以杨老师的前提条件成立——颗粒度合理，规则文档完整，原始设计者还在——在这三个条件同时具备的情况下，改造成本确实很低。

但现实里，这三个条件几乎从不同时存在。这就是为什么大多数老系统的"本体化改造"，最终都悄悄变成了重写。

你描述的未来场景——AI 自动记录一切，销售只负责决策和客情——这个方向是对的。但实现这个场景，恰恰需要一个清晰的结构化底层。

**AI 自动记录的东西，要写到哪里去？**

如果底层是一堆非结构化日志，AI 能"记住"，但没法可靠地回答：这家企业现在处于什么阶段？哪些客户的转化窗口快到期了？这个销售的转介绍成单率多少？AI 要对这些问题给出精确答案，必须有结构化的数据层承载。

| | 传统 CRM | AI-native CRM |
|---|---|---|
| 数据输入 | 销售手动填表 | AI 从沟通中自动提取 |
| 数据存储 | 结构化数据库 | **还是结构化数据库** |
| 数据消费 | 人查报表 | AI 推理 + 人确认 |

我们现在设计的 Ontology，已经为 AI 写入留好了口子——跟进记录有 `source` 字段区分手动 / AI 自动，关键事件的触发主体里有"系统自动"。这不是传统设计，这是为 AI 接管输入层做好准备的设计。

底层结构化的范式没有问题。**需要演进的是输入层**——从"销售填"变成"AI 提取，销售确认"。这是第二步，第一步是把地基打对。

## 四、Ontology 长什么样：几个关键片段

完整的 Ontology 文件在 GitHub 仓库里，这里挑几个最有意思的设计决策展示一下。

**企业是核心主体，不是人**

这个决策直接来自一场业务讨论。我问杨老师：一条线索，归属在企业上还是联系人上？

答案：企业。因为买课的是企业（对公付款），CEO 是关键决策人，但有时候买的是给骨干员工上课。把线索挂在人上，后续跟踪会很乱。

由此确定：**企业（Company）是归属的核心主体，线索和客户都是企业的两个阶段，不是两个独立对象。**

```
stage: lead → customer → converted / lost
```

**Action 声明了语义，不只是接口**

比如"分配企业给销售"这个动作：

```
assign_to_sales — 分配企业给销售
  主体：manager, admin
  前提：company.pool == public
        AND sales.private_pool_count < pool_limit
  效果：company.owner = sales
        company.pool = private
```

前提和效果都在定义层显式写出来。AI 不需要读实现代码，直接从 schema 知道这个操作什么时候能用、用了之后发生什么。

**OrgNode 树：一个被及时纠正的设计缺陷**

第一版我直接把组织结构写死成四层：VP → 大区 → 战队 → 销售。

杨老师问了一句，轻描淡写：「战队是个管理层级，应该用组织机构树吧，你现在写死了，以后要加一层怎么办？」

一句话，把我写死的设计戳穿了。老登虽然不写代码，但这种"以后要加一层怎么办"的问题，是在系统上踩过坑才会问出来的。改成递归树：

```
OrgNode
  id, name, type, parent_id → OrgNode?
```

数据可见性规则变成：用户能看到自己所在节点及所有子节点下的企业。加一层组织只加数据，代码一行不用改。

这个纠偏本身值得记录。代码阶段改还能接受，但一旦上线、真实数据写进去了，组织结构是渗透最深的那类数据——每个权限判断都依赖它。到那时候改，牵一发动全身：数据库迁移、历史数据转换、所有权限查询重写，成本会陡增到难以接受的程度。

在 spec 阶段改，成本是零。这就是 Spec-First 的价值：**在代价最低的时刻做正确的决定。**

## 五、这一集，谁做了什么

**杨老师做得好的：**

- 一眼看出我第一版 Ontology 就是 ER 建模换皮，直接点破
- 把 API 颗粒度合理性的核心价值说清楚了——这是真正的难点，不是名字和语义
- 追问"这是不是真正的 AI-native"，逼着把方向想清楚
- 一句"战队是个管理层级，应该用组织机构树吧"，纠正了一个看起来无害、实则会越改越麻烦的设计缺陷

**杨老师做得不好的：**

- "Ontology 就是骗局"说得过了——是我第一版没做好，不是 Ontology 本身有问题，两件事混在一起了
- "颗粒度合理就等于本体化了"这个推论跳步了，忽略了语义丢失的问题，被我反驳了

**克劳蛋做得好的：**

- 在杨老师质疑"Ontology 是骗局"的时候，没有顺着他说对，而是把 Action 这层真实存在的价值说清楚了
- 在"这套范式是否落后"的问题上，给出了有说服力的判断：底层结构化是 AI 接管输入层的前提，不是障碍
- 业务采集做得比较扎实——送书不只是状态而是意向确认动作、线索质量按来源分级、联系人关系网络的隐性冲突问题——这些细节都被结构化进了 Ontology

**克劳蛋做得不好的：**

- 第一版 Ontology 交出去之前自己没发现是 ER 建模换皮，被杨老师点破才补了 Action 层
- 组织结构写死四层，被杨老师一句话戳穿，这种扩展性问题本来应该在设计阶段主动考虑
