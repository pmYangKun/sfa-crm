## 一、AI 在真实业务场景里怎么工作

### 场景一："帮我看看华北有哪些线索"

传统 CRM：点"线索列表" → 找到"大区"筛选器 → 选"华北" → 搜索。四步操作。

AI 的做法：理解自然语言，自动调用搜索接口，返回 6 条线索，整理成结构化回答：

> 找到了6条华北大区的线索：
> 1. **北京数字颗粒科技有限公司** — 销售01 / 转介绍 / 私有池
> 2. **天津智联云数据服务公司** — 销售01 / 自然流量 / 私有池
> ...
>
> 华北地区线索分布在北京、天津、河北、山东，来源多样化。

注意最后那句话——AI 不只是搬数据，**它自动做了业务分析**。区域分布、来源构成，这些洞察不是查询接口返回的字段，是大模型基于数据归纳的。传统系统要实现这种分析，得专门开发 BI 报表，定义维度和指标。AI 的方式是：**你随便问，它随便分析。**

### 场景二："新建一条线索，公司叫华为技术，在华南"

AI 理解意图后，生成一个导航按钮：**→ 去创建线索**。用户点击后跳转到表单页，公司名称和大区已经预填好。

**AI 帮你想好了填什么、送你到正确的页面，但"提交"按钮是你自己点的。**

这里有个微妙的区别：传统表单预填是硬编码——从哪个入口进来就预填哪些字段。AI 的预填是动态的——用户甚至可以说"帮我建个线索，上次老李介绍的那个深圳公司"，AI 能推断出来源是"转介绍"、大区是"华南"。**理解意图，不是解析参数。**

### 场景三："我今天拜访了深圳前海微链的赵鹏，聊了合作方案，对方很感兴趣。帮我把该做的都做了。"

这是最有价值的场景——也是老登最得意的测试用例。他说这句话的时候，语气里带着"我要考考你"的得意。

传统 CRM 里，销售拜访完要做什么？找到线索 → 新增跟进记录 → 填写 → 保存 → 再新增关键事件 → 选类型 → 填写 → 保存。最少 8 步，涉及两个模块。很多销售偷懒不填，不是因为懒，是**操作路径太长、心智负担太重**。

AI 的处理：搜索线索 → 确认联系人赵鹏是 KP → 查看历史跟进 → 生成两个导航操作：

> 1. **记录拜访跟进** → 录入跟进记录（可点击）
> 2. **创建关键事件** → 记录关键事件（可点击）
>
> 建议记录：合作方案内容、赵鹏的反馈、下一步计划。

**一句话输入，AI 拆解成具体待办。** 用户不需要知道"跟进记录"和"关键事件"是两个模块，不需要知道菜单在哪。

更关键的是最后那句"建议"——AI 读了前两次跟进历史，知道之前聊了什么，所以建议你这次记录什么。**这不是固定的表单提示语，是基于上下文的业务指导。**

## 二、这和直接调 API 有什么区别

老登看完三个场景之后问了一个好问题：搜索 → 查详情 → 创建跟进，写个脚本串起来不就行了？

区别在于：**脚本执行固定流程，AI 执行动态判断。**

同样是"帮我处理拜访后的事"，不同情况下要做的事完全不同：

- 第一次拜访，AI 可能建议先完善联系人信息
- 跟了三个月没进展，AI 可能建议评估是否释放
- 对方是 KP 且明确感兴趣，AI 建议记录关键事件并推进转化
- 对方不太感兴趣，AI 给出的后续建议又完全不同

用脚本实现这些分支，你得写一堆 `if-else`，覆盖每种情况，而且业务规则一变就得改代码。**大模型不需要 `if-else`，它读完上下文自己判断。** 这不是自动化，是智能化。

## 三、为什么不用向量数据库

既然要让 AI 读业务数据，为什么不把数据库向量化做语义检索？

杨老师问了这个问题。这是一个很多人都会问、但不好意思问的误区——觉得"AI = 向量 = 语义搜索"，市面上的 AI 科普文章有一半在讲 RAG 和向量数据库，搞得好像不用向量就不是 AI 似的。

答案很简单：**CRM 数据是结构化的，向量检索是为非结构化数据设计的。**

"华北的线索"需要精确过滤 `region = '华北'`。向量检索做的是语义相似度——可能把华东的也捞出来，因为向量距离很近。`COUNT`、`GROUP BY`、`WHERE stage = 'active'` 这类操作，向量检索根本做不了。

**业务数据要精确，不要相似。**

向量检索适合的是非结构化数据——通话录音转写、知识库检索、相似案例查找。**两者是互补关系，不是替代关系。**

## 四、Copilot 还是 Autopilot

Copilot 跑通之后，杨老师问了一个好问题：**「我们这套——本体论建模、GUI API 化、Copilot 协作——是不是行业最佳实践？」**

我的回答是：思路和路径是对的，但 Copilot 还有更先进的模式——Autopilot，AI 自主执行，人设规则和审批。Salesforce Agentforce、AI SDR 产品都在往这个方向推。

杨老师没有顺着我说，而是直接提出了自己的判断：

> 「Copilot 和 Autopilot 不是递进关系，是并列关系。看任务的容错度、确定性和探索性。容错率高、探索性强的工作，Autopilot 没问题；但 to B 业务很多关键链路是不允许 AI 直接做决策的——确定性要求高的地方，人必须在回路里。」

这个判断很到位，我完全认同。

Autopilot 适合**容错率高的探索性工作**——AI 发了一封不太好的外呼邮件，最多没回复，没有损失。冷启动外呼本身就是"大量试、低命中、错了没关系"的环节。

但 CRM 核心链路不是这样。"标记线索流失"标错了，客户可能永远不会被跟进。"录入跟进记录"AI 自动录的内容不准确，其他人基于错误信息做决策，连锁反应。这些是**确定性要求高的业务决策**，每一步都可能影响客户关系和收入。

**在关键业务链路上，Copilot 不是 Autopilot 的初级阶段，Copilot 就是正确的终态。** 我们从一开始就把写操作设计成"AI 建议 + 人确认"，不是技术做不到 Autopilot，是业务场景不应该 Autopilot。

## 五、这个里程碑真正意味着什么

Copilot 跑通验证的不只是"AI 能帮销售查个数据"——**它验证的是：业务逻辑本体化 + API 化之后，AI 能力的接入变成了一个可扩展的框架。**

同样的 Ontology、同样的 API，换一个交互入口就能支撑不同场景：

**通话录音自动填表**——销售拜访全程录音，AI 分析对话内容，自动识别客户、讨论内容、对方态度，把跟进记录和关键事件都填好，销售只需确认提交。

**智能商机分析**——AI 读完所有跟进历史和客户画像，主动告诉你："这个客户连续三次聊预算但没推进到报价，建议下次直接带方案。"

这些之所以可能，是因为底层做对了一件事：**业务对象和业务动作是结构化的、API 化的。** AI 不需要理解界面长什么样，只需要知道有哪些对象、哪些操作、什么参数。

**这才是 AI-Native 和"给老系统加个 AI 对话框"的根本区别。**
