## 一、AI 怎么控制界面：五种方式，一条进化线

讨论从一个基础问题开始：AI 要操控一个软件系统，有哪些技术路线？

我列了四种。杨老师说：「你是不是没说全，我记着还有个 DOM 方式吧。」

他说得对，我漏了。补上之后，完整的五种是：

**① CLI——命令行**
AI 直接执行系统命令、读写文件。这是最底层、最稳定的控制方式，但只适合操作系统层面，不适合应用界面。

值得一提：飞书、钉钉最近都在推「CLI 化」，搞得很多人以为企业工具都得做成命令行接口才能让 AI 理解。这是误导。CLI 化有它的价值——让开发者可以通过脚本调用企业工具——但它只是五种路线里最基础的一种，既不万能，也不是 AI 接入业务系统的首选方式。适合它的场景：文件操作、系统管理、自动化脚本。不适合它的场景：面向业务用户的应用系统。

**② DOM / Accessibility Tree——解析文档结构**
AI 读取网页的 DOM 结构（页面元素的层级树），找到按钮、输入框、链接，直接操作。不靠视觉，靠结构。现在很多 AI 浏览器自动化工具走的是这条路，包括 OpenClaw——它用的是 CDP（Chrome DevTools Protocol，Chrome 内置的调试协议），本质上就是用代码读懂页面结构再操作。

**③ Computer Use——截图+模拟点击**
给 AI 看截图，让它判断点哪里，然后模拟鼠标点击。能用，但贵、慢、脆——页面稍微改个布局，AI 就找不到按钮了。是没有其他选择时的最后手段。

**④ GUI API——每个操作有对应接口**
不依赖界面，直接调后端 API 执行业务操作。「分配线索」不是「点那个按钮」，而是 `POST /leads/123/assign`。AI 和人走同一套接口，最稳、语义最清晰。

**⑤ DSL——操控底层数据模型**
专门用于画布类、工作流类工具。用户看到的是可视化流程图，AI 操控的是背后的 JSON/YAML 数据结构。Figma、飞书多维表格的 AI 功能走的是这条路。

---

从 RPA 到 OpenClaw 到 GUI API，其实是一条清晰的进化线：

> **RPA**：「记住你点了哪个坐标」——按钮挪一下，脚本就断
>
> **OpenClaw / DOM**：「读懂页面结构再点」——依赖界面语义，比 RPA 稳，但还是在解析界面
>
> **GUI API**：「根本不用点——直接调业务接口」——不依赖界面，只依赖 API 定义

## 二、老系统为什么 AI 操控不了

这个问题有两个层面，杨老师在讨论中触碰到了，我觉得值得单独说清楚。

**第一个层面：业务逻辑藏在前端代码里**

很多传统 B 端系统，「分配线索」这个按钮被点击之后，前端 JS 要做一堆判断、拼装参数、按顺序调几个底层接口——这套逻辑在前端代码里，没有对应的后端 API 端点。AI 要执行这个操作，只能模拟点击，或者把整套前端逻辑重新实现一遍。GUI API 化，就是把这些隐藏在前端的业务逻辑暴露成规范的后端 API。

**第二个层面：业务语义藏在后端代码里**

传统的数据库设计（ER 模型）只管存数据，不管「这个操作意味着什么」。你看到一张 `leads` 表，看不出「分配线索」需要什么前置条件、执行后会发生什么副作用——这些逻辑埋在后端的 Service 代码里。Ontology 的 Action 设计，就是把这些隐藏在代码里的业务规则，提升到 schema 层显式声明。

两个问题，一个本质：**传统系统的业务语义是隐式的，人能理解，AI 看不懂。AI-Native 系统要做的，是把隐式语义显式化——数据层用 Ontology，操作层用 GUI API。**

## 三、AI Agent：为什么不用 Dify

技术栈讨论进入 AI Agent 部分，杨老师说：「我本来以为要用 n8n 或者 Dify。」

这是一个很常见的判断——产品圈里很多人一提到企业 AI，第一反应就是「上 Dify」。

这个判断不能说错，但得看场景。

Dify 和 n8n 是**可视化编排平台**，把 LLM 调用、工具调用、对话记忆这些能力打包成拖拽界面，给不会写代码的人用。它的价值是降低门槛，代价是控制权。

当你的系统有自定义 Ontology、有自己的 API 层、需要 Agent 深度理解业务数据模型的时候，Dify 进来就是一个黑盒坐在你的系统旁边。它不理解你的 Ontology，工具调用要绕一圈，出了问题很难调试。

更重要的是——**Dify 做的每一件事，都有颗粒度更低的开源组件可以替代**：

| Dify 提供的能力 | 自己组装的方案 |
|---|---|
| LLM 调用 | Vercel AI SDK |
| 多模型切换 | Vercel AI SDK 配置 |
| 工具调用 | SDK 原生 Tool Use |
| 对话记忆 | 自己存数据库，几行代码 |
| 复杂流程编排 | LangGraph |

对有研发团队的产品来说，理解这一层很重要。**Dify 是平台，不是组件**——在它上面搭，灵活性上限很低。用低层组件自己组装，麻烦一点，但每一层都在你的控制里。

补充说明两个术语：Vercel AI SDK 是 Next.js 母公司出的开源库，专门用于 Web 应用接入 AI，核心价值是抹平各家 LLM 的 API 格式差异，换模型改一个配置就行；Tool Use（工具调用）是大模型的原生能力，你预先告诉 LLM「有哪些工具可以用」，LLM 自己判断什么时候调哪个——Ontology 里定义的 Action，直接翻译成 Tool Use 的工具定义，这就是前几集花时间做 Action 建模的原因。

## 四、切换 LLM 这件事

杨老师提了一个很务实的要求：系统上线之后，管理员要能切换 LLM。

这个需求合理。今天 Claude 好用，明天可能 DeepSeek 更便宜，后天可能监管要求用国产模型。技术架构上不能绑死。

实现方式：LLM Provider 和 Model Name 作为系统配置存在数据库里，Admin 后台有配置页面。

```
llm_provider = "anthropic"
llm_model    = "claude-sonnet-4-6"
```

切成 DeepSeek，改两行配置，实时生效，不需要重新部署。Vercel AI SDK 读配置动态实例化对应的 Provider，业务代码和工具定义完全不动。

这里杨老师问了一个好问题：「Skill 是 Claude 的概念么，换了其他 LLM 还有效么？」

这个问题的背后有个误解——Skill 听起来像是某家模型的专属功能。

实际上，我们设计的 Skill 就是数据库里的一段文本，LLM 通过工具调用取回来读。任何支持工具调用的 LLM 都能用，和哪家模型无关。**切换 LLM 不影响 Skill，因为 Skill 的内容存在你自己的系统里，LLM 只是读者，换个读者，文章还在。**

## 六、Skill：把最有经验的销售的判断逻辑写进系统

先说清楚一个误解。很多人听到「Skill」，脑子里浮现的是 Claude 那套规范格式——有专门的文件夹结构、SKILL.md 文件、reference 引用……搞得像一套严肃的工程规范。

这个印象让「Skill」这个词变得比实际复杂了很多。

**Skill 的本质就是提示词——一段告诉 AI 该怎么做的文本。** 没有魔法，没有特殊格式，就是写清楚：遇到这种情况，按这个逻辑来。

具体到这个 CRM：把销售团队里最有经验的人的判断逻辑，写成文本，存在数据库里，Admin 后台维护，研发不需要参与。

这个 CRM 里有用的 Skill 比如：

- **线索质量评估**：收到书了吗？回应了吗？参加过小课吗？三个信号都有的成单率高，优先跟进
- **小课后跟进 SOP**：付了小课但 14 天内未转化，第几天该说什么、什么信号可以推进
- **久未回应激活话术**：7 天没联系，推荐切入角度

Skill 有两种用法：

**用法一：用户主动问**。销售在 chat 里说「这个客户下一步该怎么做」，Agent 拿到跟进历史，检索匹配的 Skill，结合具体数据给出建议。

**用法二：系统主动推**。这是 Gong.io 做的事——系统监听到事件（新跟进录入、小课付款、14 天未跟进），Agent 自动运行，调 Skill 生成建议，直接推送给销售，甚至自动调 API 创建下一步任务。不需要人问，系统自己在跑。

两种用法的技术差异只有一个：谁触发 Agent。用法一是用户在 chat 里发消息触发，用法二是后端事件触发。

**Tool 和 Skill 的分工**：Tool 执行操作，改变系统状态；Skill 调取知识，辅助 LLM 给出更好的判断。两者组合才是完整的 Agent——**Skill 告诉 LLM 怎么想，Tool 告诉 LLM 怎么做。**

## 七、完整技术栈

讨论结束，汇总如下：

**基础层**

| 位置 | 技术 |
|---|---|
| 前端 | Next.js |
| 后端 | FastAPI（Python） |
| 数据库 | SQLite（演示用） |
| 部署 | Docker Compose + 腾讯云轻量服务器 |

**AI Agent 层**

| 位置 | 技术 |
|---|---|
| LLM 交互 | Vercel AI SDK |
| LLM Provider | 数据库配置，Admin 可切换 |
| 工具调用 | Ontology Actions → tool definitions |
| Skill 系统 | DB 存储，工具调用检索，Admin 维护 |
| 对话记忆 | DB 存每轮 messages |

**界面控制**：新建系统直接做 GUI API 化，每个 UI 操作对应一个后端 API 端点。AI 和人走同一套接口，不需要 Computer Use，不需要 DOM 解析。

**Agent 工作模式**：用户主动问（chat 触发）和系统主动推（事件触发）两种。执行分两档：低风险操作 Agent 直接调 API，高风险操作导航用户到预填表单由人确认。

- 「企业应用研发应该直接用 Dify」——Dify 适合非技术团队快速搭简单工作流，不适合需要深度集成自有数据模型的产品研发。
